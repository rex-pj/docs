Module 3: Khám Phá Azure Blob Storage

Tổng quan, tài nguyên, bảo mật và dự phòng

🌐 3.1 Tổng Quan Azure Blob Storage

Azure Blob storage là giải pháp lưu trữ đối tượng của Microsoft cho cloud. Blob storage được tối ưu hóa để lưu trữ lượng lớn dữ liệu phi cấu trúc. Dữ liệu phi cấu trúc là dữ liệu không tuân theo mô hình hoặc định nghĩa dữ liệu cụ thể, như văn bản hoặc dữ liệu nhị phân.

Blob Storage Được Thiết Kế Cho

  • Phục vụ hình ảnh hoặc tài liệu trực tiếp đến trình duyệt
  • Lưu trữ tệp để truy cập phân tán
  • Phát trực tuyến video và âm thanh
  • Ghi vào tệp log
  • Lưu trữ dữ liệu để sao lưu và khôi phục, khắc phục thảm họa và lưu trữ
  • Lưu trữ dữ liệu để phân tích bởi dịch vụ on-premises hoặc Azure-hosted

Người dùng hoặc ứng dụng client có thể truy cập các đối tượng trong Blob storage qua HTTP/HTTPS, từ bất cứ đâu trên thế giới. Đối tượng trong Blob storage có thể truy cập qua Azure Storage REST API, Azure PowerShell, Azure CLI hoặc thư viện client Azure Storage.

💡 Cấu trúc cơ bản: Tài khoản Azure Storage là container cấp cao nhất cho tất cả Azure Blob storage. Tài khoản lưu trữ cung cấp namespace duy nhất cho dữ liệu Azure Storage có thể truy cập từ bất cứ đâu trên thế giới qua HTTP hoặc HTTPS.

Loại Tài Khoản Lưu Trữ

Azure Storage cung cấp hai mức hiệu suất của tài khoản lưu trữ: standard và premium.

  • Standard: Tài khoản general-purpose v2 tiêu chuẩn - được khuyến nghị cho hầu hết các kịch bản
  • Premium: Hiệu suất cao hơn bằng solid-state drives (SSD). Có thể chọn: block blobs, page blobs hoặc file shares
Loại Tài Khoản Lưu Trữ Dịch Vụ Được Hỗ Trợ Tùy Chọn Dự Phòng Sử Dụng
Standard general-purpose v2 Blob Storage (bao gồm Data Lake Storage), Queue Storage, Table Storage, Azure Files LRS / GRS / RA-GRS / ZRS / GZRS / RA-GZRS Loại tài khoản lưu trữ tiêu chuẩn cho blobs, file shares, queues và tables. Được khuyến nghị cho hầu hết kịch bản Azure Storage.
Premium block blobs Blob Storage (bao gồm Data Lake Storage) LRS và ZRS Loại tài khoản lưu trữ premium cho block blobs và append blobs. Khuyến nghị cho kịch bản tỷ lệ giao dịch cao hoặc sử dụng đối tượng nhỏ hơn hoặc yêu cầu độ trễ lưu trữ thấp nhất quán.
Premium file shares Azure Files LRS và ZRS Loại tài khoản lưu trữ premium chỉ cho file shares. Khuyến nghị cho enterprise hoặc ứng dụng hiệu suất cao.
Premium page blobs Chỉ Page blobs LRS và ZRS Loại tài khoản lưu trữ premium chỉ cho page blobs.

Storage Accounts (Tài Khoản Lưu Trữ)

Tài khoản lưu trữ cung cấp namespace duy nhất trong Azure cho dữ liệu của bạn. Ví dụ, nếu tài khoản lưu trữ có tên mystorageaccount, endpoint mặc định cho Blob storage là:

Code
http://mystorageaccount.blob.core.windows.net

Containers (Container)

Container tổ chức một tập hợp blobs, tương tự thư mục trong hệ thống tệp. Tài khoản lưu trữ có thể chứa số lượng container không giới hạn, và container có thể lưu trữ số lượng blobs không giới hạn.

Quy tắc đặt tên container:

  • Tên container phải từ 3 đến 63 ký tự
  • Phải bắt đầu bằng chữ cái hoặc số
  • Chỉ chứa chữ thường, số và ký tự gạch ngang (-)
  • Hai hoặc nhiều ký tự gạch ngang liên tiếp không được phép

URI cho container:

Code
https://myaccount.blob.core.windows.net/mycontainer

🔐 3.3 Tính Năng Bảo Mật Azure Storage

Azure Storage sử dụng mã hóa phía dịch vụ (SSE - Service-Side Encryption) để tự động mã hóa dữ liệu khi lưu vào cloud. Mã hóa Azure Storage bảo vệ dữ liệu và giúp đáp ứng cam kết bảo mật và tuân thủ tổ chức.

Mã Hóa Dữ Liệu Tại Chỗ (Data at Rest)

  • Tự động mã hóa và giải mã dữ liệu trong Azure Storage một cách minh bạch
  • Sử dụng mã hóa AES 256-bit - một trong những mật mã khối mạnh nhất
  • Tuân thủ FIPS 140-2
  • Tương tự mã hóa BitLocker trên Windows
  • Được bật cho tất cả tài khoản lưu trữ và không thể tắt
  • Không tốn thêm chi phí

Dữ liệu được mã hóa bất kể:

  • Mức hiệu suất, access tier hoặc mô hình triển khai
  • Tất cả block blobs, append blobs và page blobs mới và hiện có
  • Khi geo-replication được bật, cả vùng chính và phụ đều được mã hóa
  • Tất cả tài nguyên Azure Storage: blobs, disks, files, queues, tables
  • Tất cả metadata đối tượng cũng được mã hóa

Quản Lý Khóa Mã Hóa

Dữ liệu trong tài khoản lưu trữ mới được mã hóa bằng Microsoft-managed keys theo mặc định.

Mã Hóa Phía Client (Client-side Encryption)

Thư viện client Azure Blob Storage cho .NET, Java và Python hỗ trợ mã hóa dữ liệu trong ứng dụng client trước khi upload lên Azure Storage và giải mã khi download.

Thư viện Blob Storage và Queue Storage sử dụng AES để mã hóa dữ liệu người dùng. Có hai phiên bản:

  • Version 2: Sử dụng chế độ Galois/Counter Mode (GCM) với AES. Blob Storage và Queue Storage SDKs hỗ trợ v2.
  • Version 1: Sử dụng chế độ Cipher Block Chaining (CBC) với AES. Blob Storage, Queue Storage và Table Storage SDKs hỗ trợ v1.

🔁 3.4 Dự Phòng Azure Blob Storage

Azure Storage luôn lưu trữ nhiều bản sao dữ liệu để bảo vệ khỏi các sự kiện có kế hoạch và không có kế hoạch. Các tùy chọn dự phòng khác nhau có sẵn:

LRS — Locally Redundant
3 bản sao trong cùng 1 data center
Bảo vệ khỏi lỗi phần cứng server/rack
Chi phí thấp nhất
Durability: 11 nines (99.999999999%)
ZRS — Zone-Redundant
3 bản sao qua 3 Availability Zones
Bảo vệ khỏi lỗi toàn bộ zone
Durability: 12 nines (99.9999999999%)
GRS — Geo-Redundant
LRS vùng chính + nhân bản sang vùng phụ (LRS)
Vùng chính và phụ cách nhau hàng trăm dặm
Durability: 16 nines (99.99999999999999%)

GRS + cho phép đọc từ vùng phụ
Durability: 16 nines

Tùy chọn dự phòng toàn diện nhất

Module 4: Quản Lý Vòng Đời Azure Blob Storage

Lifecycle management, chính sách, và triển khai

⏳ 4.1 Khám Phá Vòng Đời Blob Storage

Các tập dữ liệu có vòng đời độc đáo. Đầu vòng đời, người ta truy cập một số dữ liệu thường xuyên. Nhưng nhu cầu truy cập giảm đáng kể khi dữ liệu già đi. Một số dữ liệu không hoạt động trên cloud và hiếm khi được truy cập sau khi lưu trữ. Một số dữ liệu hết hạn sau vài ngày hoặc tháng sau khi tạo.

Access Tiers trong Quản Lý Vòng Đời

  • HOT Tier online tối ưu cho dữ liệu truy cập thường xuyên.
  • COOL Tier online tối ưu cho dữ liệu ít truy cập, lưu tối thiểu 30 ngày.
  • COLD Tier online tối ưu cho dữ liệu ít truy cập, lưu tối thiểu 90 ngày. Chi phí lưu trữ thấp hơn cool nhưng chi phí truy cập cao hơn.
  • ARCHIVE Tier ngoại tuyến tối ưu cho dữ liệu hiếm khi truy cập, lưu tối thiểu 180 ngày với yêu cầu độ trễ linh hoạt (tính bằng giờ).
💡 Quan trọng: Giới hạn lưu trữ dữ liệu được đặt ở cấp tài khoản và không phải theo access tier. Bạn có thể chọn sử dụng toàn bộ giới hạn trong một tier hoặc trên cả ba tier.

Quản Lý Vòng Đời Dữ Liệu

Azure Blob Storage lifecycle management cung cấp chính sách dựa trên quy tắc mà bạn có thể sử dụng để chuyển dữ liệu blob sang access tier thích hợp hoặc hết hạn dữ liệu vào cuối vòng đời dữ liệu.

Với chính sách lifecycle management, bạn có thể:

  • Chuyển blobs từ cool sang hot ngay khi được truy cập, để tối ưu hiệu suất
  • Chuyển phiên bản hiện tại, phiên bản trước hoặc blob snapshots sang tier lưu trữ mát hơn nếu không được truy cập hoặc sửa đổi trong một khoảng thời gian
  • Xóa phiên bản hiện tại, phiên bản trước hoặc blob snapshots vào cuối vòng đời
  • Áp dụng quy tắc cho toàn bộ tài khoản lưu trữ, container được chọn hoặc tập hợp blobs sử dụng name prefixes hoặc blob index tags làm bộ lọc
0d
HOT — Truy cập thường xuyênTạo blob, truy cập thường xuyên. Lưu trữ cao nhất, truy cập thấp nhất.
30d
COOL — Truy cập thỉnh thoảngSau ~30 ngày. Lifecycle Policy tự động chuyển sang Cool tier. Chi phí lưu trữ giảm.
90d
ARCHIVE — Lưu trữ lâu dàiSau ~90 ngày. Dữ liệu ngoại tuyến, độ trễ truy cập tính bằng giờ. Chi phí rất thấp.
7yr
DELETE — Hết hạnSau 2.555 ngày (7 năm). Policy tự động xóa blob khi hết vòng đời.
Lifecycle Policy chuyển tier tự động dựa trên: daysAfterModificationGreaterThan · daysAfterCreationGreaterThan · daysAfterLastAccessTimeGreaterThan

📋 4.2 Khám Phá Chính Sách Vòng Đời Blob Storage

Chính sách lifecycle management là tập hợp các quy tắc trong tài liệu JSON. Mỗi định nghĩa quy tắc trong chính sách bao gồm filter set và action set.

JSON
{
  "rules": [
    {
      "name": "rule1",
      "enabled": true,
      "type": "Lifecycle",
      "definition": {...}
    },
    {
      "name": "rule2",
      "type": "Lifecycle",
      "definition": {...}
    }
  ]
}

Tham Số Chính Sách (Policy)

Tên Tham Số Loại Tham Số Ghi Chú
rules Mảng rule objects Cần ít nhất một quy tắc trong chính sách. Có thể định nghĩa tối đa 100 quy tắc.

Tham Số Quy Tắc (Rule)

Tên Tham Số Loại Ghi Chú Bắt Buộc
name String Tên quy tắc tối đa 256 ký tự alphanumeric. Case-sensitive. Phải duy nhất trong chính sách. Có
enabled Boolean Tùy chọn boolean để tạm thời vô hiệu hóa quy tắc. Mặc định là true. Không
type Enum Loại hợp lệ hiện tại là Lifecycle. Có
definition Object Mỗi definition gồm filter set và action set. Có

Ví Dụ Quy Tắc Đầy Đủ

Quy tắc này lọc tài khoản để chạy các hành động trên objects trong sample-container bắt đầu với blob1:

JSON
{
  "rules": [
    {
      "enabled": true,
      "name": "sample-rule",
      "type": "Lifecycle",
      "definition": {
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            },
            "tierToArchive": {
              "daysAfterModificationGreaterThan": 90,
              "daysAfterLastTierChangeGreaterThan": 7
            },
            "delete": {
              "daysAfterModificationGreaterThan": 2555
            }
          },
          "snapshot": {
            "delete": {
              "daysAfterCreationGreaterThan": 90
            }
          }
        },
        "filters": {
          "blobTypes": [
            "blockBlob"
          ],
          "prefixMatch": [
            "sample-container/blob1"
          ]
        }
      }
    }
  ]
}

Quy tắc này:

  • Chuyển blob sang cool tier sau 30 ngày kể từ lần sửa đổi cuối
  • Chuyển blob sang archive tier sau 90 ngày kể từ lần sửa đổi cuối
  • Xóa blob sau 2.555 ngày (7 năm) kể từ lần sửa đổi cuối
  • Xóa blob snapshots sau 90 ngày kể từ khi tạo snapshot

Rule Filters (Bộ Lọc Quy Tắc)

Bộ lọc giới hạn hành động quy tắc cho tập hợp blobs trong tài khoản lưu trữ. Nếu nhiều hơn một bộ lọc được định nghĩa, logic AND chạy trên tất cả bộ lọc.

Tên Bộ Lọc Loại Bắt Buộc
blobTypes Mảng enum values được xác định trước Có
prefixMatch Mảng chuỗi cho prefix cần khớp. Mỗi quy tắc tối đa 10 prefixes. Chuỗi prefix phải bắt đầu bằng tên container. Không
blobIndexMatch Mảng dictionary values gồm điều kiện key và value của blob index tag. Mỗi quy tắc tối đa 10 điều kiện blob index tag. Không

Rule Actions (Hành Động Quy Tắc)

Hành Động Phiên Bản Hiện Tại Snapshot Phiên Bản Trước
tierToCool Hỗ trợ blockBlob Hỗ trợ Hỗ trợ
tierToCold Hỗ trợ blockBlob Hỗ trợ Hỗ trợ
enableAutoTierToHotFromCool Hỗ trợ blockBlob Không hỗ trợ Không hỗ trợ
tierToArchive Hỗ trợ blockBlob Hỗ trợ Hỗ trợ
delete Hỗ trợ blockBlob và appendBlob Hỗ trợ Hỗ trợ
⚠️ Ưu tiên hành động: Nếu bạn định nghĩa nhiều hơn một hành động trên cùng blob, lifecycle management áp dụng hành động ít tốn kém nhất. Ví dụ: delete rẻ hơn tierToArchive. tierToArchive rẻ hơn tierToCool.

Điều Kiện Chạy (Run Conditions)

Điều Kiện Chạy Giá Trị Mô Tả
daysAfterModificationGreaterThan Integer (ngày) Điều kiện cho hành động base blob. Dùng last modified time.
daysAfterCreationGreaterThan Integer (ngày) Điều kiện cho hành động blob snapshot. Dùng snapshot creation time.
daysAfterLastAccessTimeGreaterThan Integer (ngày) Điều kiện cho phiên bản hiện tại khi access tracking được bật.
daysAfterLastTierChangeGreaterThan Integer (ngày) Thời gian tối thiểu blob rehydrated được giữ ở hot/cool/cold tier trước khi trở về archive. Chỉ áp dụng cho tierToArchive.

Bạn có thể thêm, chỉnh sửa hoặc xóa chính sách bằng bất kỳ phương pháp nào sau đây:

Azure Portal - Code View

  1. Trong Azure portal, điều hướng đến tài khoản lưu trữ

Ví dụ chính sách di chuyển block blob bắt đầu với log sang cool tier nếu đã hơn 30 ngày kể từ khi blob được sửa đổi:

JSON

      "enabled": true,
      "name": "move-to-cool",

            "tierToCool": {

          }

          "blobTypes": [

        }

  ]
}

🔧 5.2 Tạo Client Object

Làm việc với bất kỳ tài nguyên Azure nào bằng SDK bắt đầu bằng việc tạo client object. Khi ứng dụng tạo client object, bạn truyền URI tham chiếu endpoint đến constructor client.

💡 Xác thực: Các ví dụ code sử dụng DefaultAzureCredential để xác thực đến Azure qua Microsoft Entra security principal. Quá trình xác thực bao gồm lấy access token để ủy quyền. Access token này được truyền như credential khi client được khởi tạo và credential tồn tại suốt vòng đời client.

Tạo BlobServiceClient Object

Object BlobServiceClient được ủy quyền cho phép ứng dụng tương tác với tài nguyên ở mức tài khoản lưu trữ. Nó cung cấp các phương thức để lấy và cấu hình thuộc tính tài khoản, cũng như liệt kê, tạo và xóa containers.

C#
using Azure.Identity;
using Azure.Storage.Blobs;

public BlobServiceClient GetBlobServiceClient(string accountName)
{
    BlobServiceClient client = new(
        new Uri($"https://{accountName}.blob.core.windows.net"),
        new DefaultAzureCredential());

    return client;
}

Tạo BlobContainerClient Object

Bạn có thể sử dụng object BlobServiceClient để tạo BlobContainerClient mới cho phép tương tác với container cụ thể:

C#
public BlobContainerClient GetBlobContainerClient(
    BlobServiceClient blobServiceClient,
    string containerName)
{
    // Tạo container client sử dụng service client object
    BlobContainerClient client = blobServiceClient.GetBlobContainerClient(containerName);
    return client;
}

Nếu công việc được giới hạn trong một container duy nhất, bạn có thể tạo BlobContainerClient trực tiếp:

C#
public BlobContainerClient GetBlobContainerClient(
    string accountName,
    string containerName,
    BlobClientOptions clientOptions)
{
    // Thêm tên container vào cuối URI
    BlobContainerClient client = new(
        new Uri($"https://{accountName}.blob.core.windows.net/{containerName}"),
        new DefaultAzureCredential(),
        clientOptions);

    return client;
}

Tạo BlobClient Object

Để tương tác với tài nguyên blob cụ thể, tạo BlobClient object từ service client hoặc container client:

C#
public BlobClient GetBlobClient(
    BlobServiceClient blobServiceClient,
    string containerName,
    string blobName)
{
    BlobClient client =
        blobServiceClient.GetBlobContainerClient(containerName).GetBlobClient(blobName);
    return client;
}

→ .GetBlobClient(blobName)
Tùy Chọn 2 — Trực tiếp Container
BlobContainerClient(Uri, Credential, Options)
→ .GetBlobClient(blobName)
Tùy Chọn 3 — Trực tiếp Blob
BlobClient(Uri, Credential)
Dùng khi chỉ làm việc với 1 blob cụ thể
Auth & URI Format
Xác thực: DefaultAzureCredential() (Microsoft Entra)
URI: https://{account}.blob.core.windows.net/{container}/{blob}

🏷️ 5.3 Quản Lý Thuộc Tính Container và Metadata bằng .NET

Blob containers hỗ trợ system properties và user-defined metadata, ngoài dữ liệu chúng chứa.

  • System properties (Thuộc tính hệ thống): Tồn tại trên mỗi tài nguyên Blob storage. Một số có thể đọc hoặc đặt, một số chỉ đọc. Một số system properties tương ứng với các HTTP header tiêu chuẩn. Thư viện client Azure Storage cho .NET duy trì các thuộc tính này cho bạn.
  • User-defined metadata (Metadata do người dùng định nghĩa): Gồm một hoặc nhiều cặp name-value mà bạn chỉ định cho tài nguyên Blob storage. Bạn có thể sử dụng metadata để lưu trữ các giá trị bổ sung. Giá trị metadata chỉ dành cho mục đích của bạn và không ảnh hưởng đến hành vi tài nguyên.
⚠️ Quy tắc đặt tên Metadata: Cặp name/value metadata là HTTP headers hợp lệ, vì vậy phải tuân thủ tất cả hạn chế của HTTP headers. Tên metadata phải là HTTP header names hợp lệ và C# identifiers hợp lệ, chỉ chứa ASCII characters, và nên được xử lý case-insensitive. Giá trị metadata chứa non-ASCII characters nên được Base64-encoded hoặc URL-encoded.

Lấy Thuộc Tính Container

Để lấy thuộc tính container, gọi một trong các phương thức sau của BlobContainerClient:

  • GetProperties
  • GetPropertiesAsync
C#
private static async Task ReadContainerPropertiesAsync(BlobContainerClient container)
{
    try
    {
        // Lấy một số thuộc tính container và ghi giá trị ra console
        var properties = await container.GetPropertiesAsync();
        Console.WriteLine($"Properties for container {container.Uri}");
        Console.WriteLine($"Public access level: {properties.Value.PublicAccess}");
        Console.WriteLine($"Last modified time in UTC: {properties.Value.LastModified}");
    }
    catch (RequestFailedException e)
    {
        Console.WriteLine($"HTTP error code {e.Status}: {e.ErrorCode}");
        Console.WriteLine(e.Message);
        Console.ReadLine();
    }
}

Đặt và Lấy Metadata

Để đặt metadata, thêm cặp name-value vào object IDictionary, sau đó gọi một trong các phương thức sau:

  • SetMetadata
  • SetMetadataAsync
⚠️ Lưu ý: Nếu hai hoặc nhiều metadata headers với cùng tên được gửi cho một tài nguyên, Blob service trả về status code 400 (Bad Request).

Đặt metadata trên container:

C#
public static async Task AddContainerMetadataAsync(BlobContainerClient container)
{
    try
    {
        IDictionary<string, string> metadata =
           new Dictionary<string, string>();

        // Thêm một số metadata vào container
        metadata.Add("docType", "textDocuments");
        metadata.Add("category", "guidance");

        // Đặt metadata của container
        await container.SetMetadataAsync(metadata);
    }
    catch (RequestFailedException e)
    {
        Console.WriteLine($"HTTP error code {e.Status}: {e.ErrorCode}");
        Console.WriteLine(e.Message);
        Console.ReadLine();
    }
}

Lấy metadata từ container:

C#
public static async Task ReadContainerMetadataAsync(BlobContainerClient container)
{
    try
    {
        // GetPropertiesAsync cũng lấy metadata
        var properties = await container.GetPropertiesAsync();

        // Liệt kê metadata của container
        Console.WriteLine("Container metadata:");
        foreach (var metadataItem in properties.Value.Metadata)
        {
            Console.WriteLine($"\tKey: {metadataItem.Key}");
            Console.WriteLine($"\tValue: {metadataItem.Value}");
        }
    }
    catch (RequestFailedException e)
    {
        Console.WriteLine($"HTTP error code {e.Status}: {e.ErrorCode}");
        Console.WriteLine(e.Message);
        Console.ReadLine();
    }
}

🌐 5.4 Đặt và Lấy Thuộc Tính và Metadata qua REST

Containers và blobs hỗ trợ metadata tùy chỉnh, được biểu diễn như HTTP headers. Metadata headers có thể được đặt trên request tạo container hoặc blob mới, hoặc trên request tạo rõ ràng thuộc tính trên tài nguyên hiện có.

Định Dạng Metadata Header

Metadata headers là cặp name/value. Định dạng cho header là:

Code
x-ms-meta-name:string-value
💡 Quy tắc về tên Metadata:
  • Từ phiên bản 2009-09-19, tên metadata phải tuân thủ quy tắc đặt tên cho C# identifiers
  • Tên là case-insensitive
  • Tên metadata bảo tồn chữ hoa/thường khi được tạo nhưng case-insensitive khi đặt hoặc đọc
  • Nếu hai hoặc nhiều metadata headers với cùng tên được gửi, Blob service trả về status code 400 (Bad Request)
  • Tổng kích thước của tất cả cặp metadata tối đa là 8 KB

Các Thao Tác trên Metadata

Metadata trên blob hoặc container có thể được lấy hoặc đặt trực tiếp mà không trả về hoặc thay đổi nội dung tài nguyên. Giá trị metadata chỉ có thể đọc hoặc ghi đầy đủ; cập nhật một phần không được hỗ trợ.

Lấy Thuộc Tính và Metadata qua REST

Thao tác GET/HEAD lấy metadata headers cho container hoặc blob được chỉ định. Các thao tác này chỉ trả về headers, không trả về response body.

URI syntax để lấy metadata trên container:

HTTP
GET/HEAD https://myaccount.blob.core.windows.net/mycontainer?restype=container&comp=metadata

URI syntax để lấy metadata trên blob:

HTTP
GET/HEAD https://myaccount.blob.core.windows.net/mycontainer/myblob?comp=metadata

Đặt Metadata Headers qua REST

Thao tác PUT đặt metadata headers trên container hoặc blob được chỉ định, ghi đè bất kỳ metadata hiện có nào. Gọi PUT không có headers nào trên request sẽ xóa tất cả metadata hiện có trên tài nguyên.

URI syntax để đặt metadata trên container:

HTTP
PUT https://myaccount.blob.core.windows.net/mycontainer?comp=metadata&restype=container

URI syntax để đặt metadata trên blob:

HTTP
PUT https://myaccount.blob.core.windows.net/mycontainer/myblob?comp=metadata

Standard HTTP Properties cho Containers và Blobs

Containers và blobs cũng hỗ trợ các standard HTTP properties. Properties và metadata đều được biểu diễn như standard HTTP headers; sự khác biệt là trong cách đặt tên: metadata headers có tiền tố x-ms-meta-, còn property headers sử dụng tên HTTP header tiêu chuẩn.

Containers ETag, Last-Modified
Blobs ETag, Last-Modified, Content-Length, Content-Type, Content-MD5, Content-Encoding, Content-Language, Cache-Control, Origin, Range

Tóm Tắt Nhanh - Điểm Ôn Thi AZ-204

Điểm chính cần nhớ cho kỳ thi

🎯 Điểm Chính Cần Nhớ

Về Blob Storage Cơ Bản

  • 3 loại tài nguyên: Storage Account → Container → Blob
  • 3 loại blobs: Block (190.7 TiB), Append (logging), Page (8 TB, VHD)
  • Quy tắc container name: 3-63 chars, chữ thường/số/gạch ngang, không 2 gạch ngang liên tiếp
  • Endpoint format: https://{account}.blob.core.windows.net/{container}/{blob}

Về Access Tiers

  • Hot: Thường xuyên, lưu trữ cao nhất, truy cập thấp nhất, mặc định
  • Cool: Ít hơn, min 30 ngày, lưu trữ thấp hơn Hot
  • Cold: Hiếm hơn, min 90 ngày, lưu trữ thấp hơn Cool
  • Archive: Chỉ offline, min 180 ngày, độ trễ tính bằng giờ, rẻ nhất

Về Lifecycle Management

  • Chính sách = tập hợp rules trong JSON
  • Tối đa 100 rules mỗi chính sách
  • Filters: blobTypes (bắt buộc), prefixMatch (max 10), blobIndexMatch (max 10 tags)
  • Ưu tiên hành động: delete < tierToArchive < tierToCool (rẻ hơn = được ưu tiên)
  • Cập nhật chính sách: chỉ đọc/ghi toàn bộ, không hỗ trợ partial update

Về .NET SDK Client Library

  • SDK v12.x: BlobServiceClient, BlobContainerClient, BlobClient
  • Xác thực: DefaultAzureCredential (Microsoft Entra)
  • BlobServiceClient: top-level, quản lý tài khoản và containers
  • BlobContainerClient: quản lý container và blobs trong đó
  • BlobClient: thao tác blob cụ thể

Về Properties và Metadata

  • System properties: chuẩn HTTP, SDK duy trì tự động
  • User metadata: cặp name-value tùy ý, tối đa 8 KB
  • Metadata format: x-ms-meta-{name}:{value}
  • Đặt metadata với SetMetadataAsync, lấy với GetPropertiesAsync
  • REST: GET metadata = ?comp=metadata, PUT metadata ghi đè toàn bộ

Về Bảo Mật

  • Mã hóa AES-256 tự động, không thể tắt, không tốn thêm tiền
  • 3 loại key management: Microsoft-managed, Customer-managed, Customer-provided
  • Customer-managed keys: phải lưu trong Azure Key Vault hoặc Key Vault HSM
  • Client-side encryption: v2 (GCM) hoặc v1 (CBC) với AES

Về Dự Phòng

  • LRS: 3 bản sao trong 1 DC, chi phí thấp nhất
  • ZRS: 3 bản sao qua 3 zones, bảo vệ khỏi lỗi zone
  • GRS/RA-GRS: LRS + nhân bản vùng thứ hai (RA = có thể đọc)
  • GZRS/RA-GZRS: ZRS + nhân bản vùng thứ hai (tốt nhất)
