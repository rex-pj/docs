Module 1: Azure Functions — Serverless Computing

Tổng quan, hosting plans và cơ chế scaling

⚡ 2.1 Tổng Quan Azure Functions

Azure Functions là một giải pháp serverless (không cần máy chủ) cho phép bạn viết ít code hơn, duy trì ít cơ sở hạ tầng hơn và tiết kiệm chi phí. Thay vì lo lắng về việc triển khai và duy trì máy chủ, hạ tầng đám mây cung cấp tất cả các tài nguyên cần thiết để giữ ứng dụng của bạn hoạt động.

Chúng ta thường xây dựng các hệ thống để phản ứng với một loạt sự kiện quan trọng. Dù bạn đang xây dựng một Web API, phản hồi thay đổi cơ sở dữ liệu, xử lý luồng dữ liệu IoT, hay thậm chí quản lý hàng đợi tin nhắn — mọi ứng dụng đều cần một cách để chạy code khi các sự kiện này xảy ra.

💡 Khái Niệm Cốt Lõi
Azure Functions hỗ trợ Trigger — cách bắt đầu thực thi function — và Binding — cách kết nối dữ liệu đầu vào/đầu ra mà không cần code thủ công.

2

3

4

5

🔄 2.2 So Sánh Functions và Logic Apps

Cả Functions và Logic Apps đều là dịch vụ Azure cho phép các khối lượng công việc serverless. Azure Functions là dịch vụ tính toán serverless, trong khi Azure Logic Apps là nền tảng tích hợp quy trình làm việc (workflow) serverless. Cả hai đều có thể tạo ra các orchestration (điều phối) phức tạp — là tập hợp các hàm hoặc bước được thực thi để hoàn thành một tác vụ phức tạp.

📌 Điểm Khác Biệt Quan Trọng
  • Với Azure Functions: bạn phát triển orchestration bằng cách viết code và sử dụng Durable Functions extension.
  • Với Logic Apps: bạn tạo orchestration bằng GUI (giao diện đồ họa) hoặc chỉnh sửa các file cấu hình.

⚖️ 2.3 So Sánh Functions và WebJobs

Giống như Azure Functions, Azure App Service WebJobs với WebJobs SDK là một dịch vụ tích hợp code-first được thiết kế cho các nhà phát triển. Cả hai đều được xây dựng trên Azure App Service và hỗ trợ các tính năng như tích hợp source control, xác thực và giám sát với Application Insights.

Azure Functions được xây dựng trên WebJobs SDK, vì vậy nó chia sẻ nhiều event trigger và kết nối tới các dịch vụ Azure giống nhau. Dưới đây là các yếu tố cần xem xét khi lựa chọn giữa Azure Functions và WebJobs với WebJobs SDK:

Định giá trả theo sử dụng (Pay-per-use) ✅ Có ❌ Không
Tích hợp với Logic Apps ✅ Có ❌ Không
Sự kiện trigger Timer, Azure Storage queues và blobs, Azure Service Bus queues và topics, Azure Cosmos DB, Azure Event Hubs, HTTP/WebHook (GitHub, Slack), Azure Event Grid Timer, Azure Storage queues và blobs, Azure Service Bus queues và topics, Azure Cosmos DB, Azure Event Hubs, File system
✅ Khuyến Nghị
Azure Functions cung cấp năng suất cho nhà phát triển cao hơn Azure App Service WebJobs, đồng thời cũng cung cấp nhiều lựa chọn hơn về ngôn ngữ lập trình, môi trường phát triển, tích hợp dịch vụ Azure và định giá. Trong hầu hết các trường hợp, đây là lựa chọn tốt nhất.

Cơ sở hạ tầng Azure App Service hỗ trợ hosting Azure Functions trên cả máy ảo Linux và Windows. Tùy chọn hosting bạn chọn quyết định các hành vi sau:

  • Cách function app được co giãn (scaled)
  • Các tài nguyên có sẵn cho mỗi instance của function app
  • Hỗ trợ cho các tính năng nâng cao như kết nối Azure Virtual Network
  • Hỗ trợ cho Linux containers

📋 Chi Tiết Từng Plan

💚 Consumption Plan (Gói Tiêu Thụ)

Consumption plan là hosting plan mặc định. Thanh toán cho tài nguyên tính toán chỉ khi các function của bạn đang chạy (pay-as-you-go) với tự động co giãn. Trên Consumption plan, các instance của Functions host được thêm vào và xóa đi một cách linh hoạt dựa trên số lượng sự kiện đến.

Khi nào nên dùng: Workload không liên tục, muốn chi phí tối thiểu khi không có lưu lượng truy cập.

💙 Flex Consumption Plan (Gói Tiêu Thụ Linh Hoạt)

Có khả năng co giãn cao với lựa chọn tính toán, kết nối mạng ảo và thanh toán theo sử dụng. Trên Flex Consumption plan, các instance được thêm/xóa dựa trên concurrency được cấu hình mỗi instance và số lượng sự kiện đến.

Bạn có thể giảm cold start bằng cách chỉ định số instance được pre-provisioned (luôn sẵn sàng). Tự động co giãn dựa trên nhu cầu.

🟠 Premium Plan (Gói Cao Cấp)

Tự động co giãn dựa trên nhu cầu sử dụng prewarmed workers (worker được làm ấm sẵn), chạy ứng dụng không có độ trễ sau khi bị idle, chạy trên các instance mạnh hơn và kết nối với virtual network.

Hãy xem xét Azure Functions Premium plan trong các tình huống sau:

  • Function app của bạn chạy liên tục hoặc gần như liên tục
  • Bạn muốn kiểm soát nhiều hơn với các instance và muốn triển khai nhiều function app trên cùng một plan với event-driven scaling
  • Bạn có số lượng thực thi nhỏ cao và hóa đơn thực thi cao, nhưng GB-seconds thấp trong Consumption plan
  • Bạn cần nhiều tùy chọn CPU hoặc bộ nhớ hơn so với consumption plans
  • Code của bạn cần chạy lâu hơn thời gian thực thi tối đa cho phép trên Consumption plan
  • Bạn yêu cầu kết nối virtual network
  • Bạn muốn cung cấp custom Linux image để chạy các function của mình
⚫ Dedicated Plan (Gói Chuyên Dụng)

Chạy các function trong một App Service plan theo mức giá App Service plan thông thường. Tốt nhất cho các kịch bản chạy dài mà Durable Functions không thể sử dụng.

Hãy xem xét App Service plan trong các tình huống sau:

  • Bạn phải có hóa đơn hoàn toàn có thể dự đoán, hoặc bạn cần tự thủ công co giãn các instance
  • Bạn muốn chạy nhiều web app và function app trên cùng một plan
  • Bạn cần truy cập vào các tùy chọn kích thước tính toán lớn hơn
  • Isolation tính toán đầy đủ và truy cập mạng an toàn được cung cấp bởi App Service Environment (ASE)
  • Sử dụng bộ nhớ cao và co giãn cao (ASE)
🟣 Container Apps (Ứng Dụng Container)

Tạo và triển khai các function app được container hóa trong môi trường hoàn toàn được quản lý, được lưu trữ bởi Azure Container Apps.

Sử dụng mô hình lập trình Azure Functions để xây dựng các function app event-driven, serverless, cloud native. Chạy các function của bạn cùng với các microservice, API, website và workflow khác dưới dạng các chương trình được container-hosted.

Hãy xem xét hosting function trên Container Apps trong các tình huống sau:

  • Bạn muốn đóng gói các thư viện tùy chỉnh với code function để hỗ trợ các line-of-business app
  • Bạn cần di chuyển thực thi code từ on-premises hoặc legacy app sang cloud native microservices chạy trong container
  • Bạn muốn tránh overhead và sự phức tạp của việc quản lý Kubernetes cluster và dedicated compute
  • Bạn cần sức mạnh xử lý cao cấp được cung cấp bởi dedicated CPU compute resources cho các function của mình

⏱️ 3.1 Thời Gian Chờ (Function App Timeout Duration)

Thuộc tính functionTimeout trong file project host.json chỉ định thời gian chờ cho các function trong một function app. Thuộc tính này áp dụng cụ thể cho việc thực thi function. Sau khi trigger bắt đầu thực thi function, function cần return/respond trong thời gian chờ.

Plan Mặc định (Default) Tối đa (Maximum)
Flex Consumption plan 30 phút Không giới hạn ²
Premium plan 30 phút ⁴ Không giới hạn ²
Dedicated plan 30 phút ⁴ Không giới hạn ³
Container Apps 30 phút Không giới hạn ⁵
Consumption plan 5 phút 10 phút
⚠️ Lưu Ý Quan Trọng
  1. Bất kể cài đặt timeout của function app, 230 giây là lượng thời gian tối đa mà một HTTP triggered function có thể mất để phản hồi một yêu cầu. Đây là do idle timeout mặc định của Azure Load Balancer. Với thời gian xử lý dài hơn, hãy xem xét sử dụng Durable Functions async pattern hoặc trì hoãn công việc thực sự và trả về phản hồi ngay lập tức.
  2. Không có thời gian thực thi tối đa nào được thực thi. Tuy nhiên, grace period cho một lần thực thi function là 60 phút trong quá trình scale in cho Flex Consumption và Premium plans, và 10 phút trong quá trình cập nhật platform.
  3. Yêu cầu App Service plan được đặt thành Always On. Grace period 10 phút được cho trong quá trình cập nhật platform.
  4. Timeout mặc định cho phiên bản 1.x của Functions host runtime là không giới hạn.
  5. Khi số lượng replica tối thiểu được đặt thành zero, timeout mặc định phụ thuộc vào các trigger cụ thể được sử dụng trong app.

⚙️ 2. Tổng Quan Phát Triển Azure Functions

Một function app (ứng dụng hàm) cung cấp ngữ cảnh thực thi trong Azure nơi các function của bạn chạy. Vì vậy, nó là đơn vị triển khai và quản lý cho các function của bạn. Một function app bao gồm một hoặc nhiều function riêng lẻ được quản lý, triển khai và co giãn cùng nhau. Tất cả các function trong một function app chia sẻ cùng pricing plan, phương thức triển khai và phiên bản runtime.

📝 Lưu Ý
Trong Functions 2.x, tất cả các function trong một function app phải được viết bằng cùng một ngôn ngữ. Trong các phiên bản cũ hơn của Azure Functions runtime, điều này không bắt buộc.

💡 Tư Duy Đúng Về Function App

Hãy nghĩ về function app như một cách để tổ chức và quản lý các function của bạn một cách tập thể. Tất cả function trong một app sẽ cùng:

  • Chia sẻ pricing plan (gói giá)
  • Chia sẻ deployment method (phương thức triển khai)
  • Chia sẻ runtime version (phiên bản runtime)
  • Được scaled cùng nhau (co giãn cùng nhau)

💻 Phát Triển và Kiểm Thử Cục Bộ

Functions giúp dễ dàng sử dụng trình soạn thảo code và công cụ phát triển yêu thích của bạn để tạo và kiểm thử function trên máy tính cục bộ. Các function cục bộ của bạn có thể kết nối với các dịch vụ Azure trực tiếp, và bạn có thể debug chúng trên máy tính cục bộ sử dụng Functions runtime đầy đủ.

⚠️ Khuyến Nghị Phát Triển
Do các giới hạn khi chỉnh sửa code function trong Azure portal, bạn nên phát triển các function của mình cục bộ và publish code project lên một function app trong Azure. Không nên dùng portal để chỉnh sửa code production.

📁 2.1 Các Tệp Dự Án Cục Bộ (Local Project Files)

Một thư mục dự án Functions chứa các file sau trong thư mục gốc của dự án, bất kể ngôn ngữ:

📄 host.json

File metadata chứa các tùy chọn cấu hình ảnh hưởng đến tất cả các function trong một function app instance. Cấu hình liên quan đến bindings trong host.json được áp dụng đồng đều cho mỗi function trong function app.

Bạn cũng có thể ghi đè hoặc áp dụng cài đặt theo môi trường bằng cách sử dụng application settings.

📄 local.settings.json

Lưu trữ app settings và cài đặt được sử dụng bởi local development tools. Cài đặt trong local.settings.json chỉ được sử dụng khi bạn chạy project cục bộ.

Khi publish project lên Azure, hãy đảm bảo thêm bất kỳ cài đặt cần thiết nào vào app settings của function app.

🚨 Quan Trọng - Bảo Mật
Vì local.settings.json có thể chứa các bí mật, chẳng hạn như connection strings, bạn KHÔNG BAO GIỜ nên lưu trữ nó trong remote repository (như GitHub, Azure DevOps, v.v.). Hãy thêm file này vào .gitignore.

🔄 Đồng Bộ Cài Đặt (Synchronize Settings)

Khi bạn phát triển các function cục bộ, bất kỳ cài đặt cục bộ nào được yêu cầu bởi ứng dụng của bạn cũng phải có trong app settings của function app đã triển khai. Bạn cũng có thể tải xuống các cài đặt hiện tại từ function app về dự án cục bộ của mình.

🔌 3. Triggers & Bindings (Trình Kích Hoạt & Liên Kết)

Một Trigger xác định cách một function được gọi — mỗi function có đúng một trigger. Trigger mang dữ liệu đi kèm, thường là payload của function.

Binding là cách khai báo kết nối với tài nguyên bên ngoài; có thể là input binding, output binding, hoặc cả hai. Dữ liệu được truyền vào/ra function dưới dạng parameters.

✅ Lợi Ích của Triggers & Bindings
Trigger và binding cho phép bạn tránh hardcode quyền truy cập vào các dịch vụ khác. Function của bạn nhận dữ liệu (ví dụ: nội dung của một queue message) trong các tham số function. Bạn gửi dữ liệu (ví dụ: để tạo một queue message) bằng cách sử dụng giá trị trả về của function.
TRIGGER (bắt buộc, đúng 1)
HTTP · Timer · Queue · Blob · Event Hub · Cosmos DB · Service Bus
Direction: in
INPUT BINDING (tùy chọn, nhiều)
Blob Storage · Table Storage · Cosmos DB · SignalR · v.v.
Direction: in
FUNCTION CODE
Logic nghiệp vụ của bạn. Nhận dữ liệu từ trigger và input bindings qua tham số. Gửi kết quả qua output bindings.
OUTPUT BINDING (tùy chọn, nhiều)
Queue message · Blob · Cosmos DB · Email · Table · SignalR · v.v.
Direction: out
Quy tắc: Mỗi function PHẢI CÓ ĐÚNG 1 trigger · Input & Output binding là tùy chọn, có thể có nhiều · Một số binding đặc biệt hỗ trợ direction inout qua Advanced editor.

🧪 Kiểm Thử Cục Bộ với Triggers & Bindings

Khi phát triển function cục bộ, cần xem xét cách xử lý trigger và binding:

  • HTTP trigger: Có thể gọi HTTP endpoint trực tiếp trên máy cục bộ: http://localhost/
  • Non-HTTP triggered functions: Có nhiều tùy chọn:
    • Dùng connection strings trỏ đến dịch vụ Azure thực — thêm vào mảng Values trong local.settings.json. Lưu ý: đây là dữ liệu thực nên nên dùng dịch vụ riêng cho dev/test.
    • Với storage-based trigger: Dùng Azurite emulator cục bộ để kiểm thử Azure Storage bindings (Queue, Blob, Table) mà không cần kết nối remote storage.
    • Tự thủ công chạy non-HTTP trigger functions bằng cách sử dụng các administrator endpoints đặc biệt.

📋 Hướng Dẫn Binding Direction (Chiều Binding)

Loại Direction (Chiều) Mô Tả

Input binding in Nhận dữ liệu từ dịch vụ bên ngoài
Output binding out Gửi dữ liệu ra dịch vụ bên ngoài

📝 3.1 Định Nghĩa Binding Theo Ngôn Ngữ

Trigger và binding được định nghĩa khác nhau tùy theo ngôn ngữ phát triển và runtime model:

Ngôn Ngữ Cách Cấu Hình Trigger & Binding
C# class library Trang trí (decorating) các method và parameter với C# attributes (in-process hoặc isolated worker)
Java Trang trí các method và parameter với Java annotations
JavaScript/TypeScript v4: định nghĩa input/output trong code với @azure/functions; v3: cấu hình trong function.json mỗi function

📌 Mô Hình Lập Trình Mới vs. Cũ
  • Mô hình mới (Node.js v4, Python v2): Bạn viết cấu hình trigger và binding trong code — runtime tự động tạo ra function.json tương ứng.
  • Mô hình cũ (Node.js v3, Python v1, PowerShell): Sử dụng function.json trực tiếp.
  • Quan trọng: Bạn không thể trộn lẫn các mô hình lập trình trong cùng một function app.

    "name": "req",

Các tùy chọn cho dataType: binary, stream, string.

🔵 C# (Isolated Worker Model)

Ví dụ này hiển thị một HTTP-triggered function viết tin nhắn vào Storage Queue bằng output binding được định nghĩa bởi attributes:

C#
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Http;

public static class HttpToQueue
{
    [Function("HttpToQueue")]
    public static MultiResponse Run(
        [HttpTrigger(AuthorizationLevel.Function, "get", "post")] HttpRequestData req)
    {
        var message = "Processed request";
        return new MultiResponse
        {
            Messages = new[] { message },
            HttpResponse = req.CreateResponse(System.Net.HttpStatusCode.OK)
        };
    }
}

public class MultiResponse
{
    [QueueOutput("outqueue", Connection = "AzureWebJobsStorage")]
    public string[] Messages { get; set; }
    public HttpResponseData HttpResponse { get; set; }
}

🟡 Node.js (v4 Programming Model)

Trong mô hình lập trình Node.js v4, bạn cấu hình input và output trong code bằng cách sử dụng @azure/functions:

Python
import { app, output } from "@azure/functions";

const queueOutput = output.storageQueue({
  queueName: "outqueue",
  connection: "AzureWebJobsStorage"
});

app.http("HttpToQueue", {
  methods: ["GET", "POST"],
  authLevel: "function",
  extraOutputs: [queueOutput],
  handler: async (request, context) => {
    const body = await request.text();
    context.extraOutputs.set(queueOutput, body || "Processed request");
    return { status: 200, body: "Queued" };
  }
});

🐍 Python (v2 Programming Model)

Trong mô hình lập trình Python v2, bạn sử dụng decorators để định nghĩa bindings. Runtime tự động tạo ra function.json cho bạn:

Python
import azure.functions as func

app = func.FunctionApp()

@app.route(route="HttpToQueue", auth_level=func.AuthLevel.FUNCTION)
@app.queue_output(arg_name="msg", queue_name="outqueue", connection="AzureWebJobsStorage")
def HttpToQueue(req: func.HttpRequest, msg: func.Out[str]) -> func.HttpResponse:
    body = req.get_body().decode("utf-8") if req.get_body() else "Processed request"
    msg.set(body)
    return func.HttpResponse("Queued", status_code=200)
💡 Lưu Ý Quan Trọng cho Node.js v4 và Python v2
Trong Node.js v4 và Python v2, runtime tạo ra function.json từ code của bạn. Tránh chỉnh sửa function.json trực tiếp trong portal cho các mô hình này; hãy thực hiện thay đổi trong code và republish.

🔐 4. Kết Nối Dịch Vụ Azure (Connect Functions to Azure Services)

Là một thực hành bảo mật tốt nhất, Azure Functions tận dụng tính năng application settings của Azure App Service để giúp bạn lưu trữ an toàn hơn các chuỗi kết nối, khóa và các token khác cần thiết để kết nối với các dịch vụ khác.

Application settings trong Azure được lưu trữ mã hóa (encrypted) và được truy cập tại runtime bởi ứng dụng của bạn như các cặp name/value của biến môi trường. Đối với trigger và binding yêu cầu thuộc tính connection, bạn đặt tên của application setting thay vì connection string thực tế. Bạn không thể cấu hình binding trực tiếp với connection string hoặc key.

🔑 Cách Hoạt Động
  • Application settings được lưu trữ mã hóa trong Azure
  • Được truy cập tại runtime như biến môi trường
  • Khi chạy trong Azure: đọc từ application settings
  • Khi phát triển cục bộ: đọc từ local.settings.json
  • Default configuration provider sử dụng environment variables

🛡️ 4.1 Kết Nối Dựa Trên Danh Tính (Identity-Based Connection)

Một số kết nối trong Azure Functions được cấu hình để sử dụng danh tính (identity) thay vì bí mật (secret). Hỗ trợ phụ thuộc vào extension sử dụng kết nối. Trong một số trường hợp, connection string vẫn có thể được yêu cầu trong Functions dù dịch vụ bạn đang kết nối hỗ trợ identity-based connections.

⚠️ Ngoại Lệ Quan Trọng
Một app chạy trong Consumption hoặc Elastic Premium plan sử dụng các setting WEBSITE_AZUREFILESCONNECTIONSTRING và WEBSITE_CONTENTSHARE khi kết nối với Azure Files trên storage account được sử dụng bởi function app. Azure Files không hỗ trợ sử dụng managed identity khi truy cập file share.

🔑 Cách Identity-Based Connection Hoạt Động

Ngữ Cảnh Danh Tính Được Sử Dụng Ghi Chú
Hosted trong Azure Functions service Managed Identity System-assigned identity được dùng mặc định
User-assigned identity Chỉ định qua credential và clientID Cấu hình với resource ID không được hỗ trợ
Local development Developer identity Danh tính của nhà phát triển được dùng thay thế (có thể tùy chỉnh)

🔐 Cấp Quyền Cho Danh Tính (Grant Permission to Identity)

Các danh tính phải có quyền để thực hiện các hành động dự định. Điều này thường được thực hiện bằng cách:

  • Gán vai trò (Role Assignment) trong Azure Role-Based Access Control (RBAC)
  • Chỉ định danh tính trong access policy tùy thuộc vào dịch vụ bạn đang kết nối
🚨 Nguyên Tắc Bảo Mật Quan Trọng: Least Privilege
Dịch vụ đích có thể expose một số quyền không cần thiết cho tất cả các ngữ cảnh. Khi có thể, hãy tuân thủ nguyên tắc quyền tối thiểu (principle of least privilege) — chỉ cấp cho danh tính các quyền bắt buộc cần thiết.

  • Không cần lưu trữ và quản lý secret
  • Tự động luân phiên token (token rotation)
  • An toàn hơn — không có rủi ro lộ key
