Phần 1: Azure Event Grid — Pub/Sub Serverless

Dịch vụ pub/sub serverless theo sự kiện

Azure Event Grid là dịch vụ Pub/Sub serverless, khả năng mở rộng cao, hỗ trợ HTTP và MQTT. Với Azure Event Grid, bạn có thể:

  • 📨 Xây dựng data pipelines với dữ liệu thiết bị
  • 🔗 Tích hợp ứng dụng
  • ⚡ Xây dựng kiến trúc serverless hướng sự kiện
  • 🌐 Hỗ trợ MQTT v3.1.1 và v5.0 cho IoT
  • ☁️ Hỗ trợ CloudEvents 1.0 specification

🧩 Các Khái Niệm Trong Azure Event Grid

1. Publishers (Nhà xuất bản)

Publisher là ứng dụng gửi sự kiện đến Event Grid. Có thể là cùng ứng dụng nơi sự kiện bắt nguồn, hoặc nguồn sự kiện. Các dịch vụ Azure xuất bản sự kiện để thông báo về sự kiện xảy ra trong dịch vụ của chúng.

Partner (Đối tác): Là loại publisher gửi sự kiện từ hệ thống của họ để cung cấp cho khách hàng Azure. Partners không chỉ xuất bản mà còn có thể nhận sự kiện từ Event Grid thông qua tính năng Partner Events.

2. Events và CloudEvents (Sự kiện)

Sự kiện là lượng thông tin nhỏ nhất mô tả đầy đủ điều gì đó đã xảy ra. Mỗi sự kiện có thông tin chung như:

  • source - nguồn của sự kiện
  • time - thời điểm xảy ra sự kiện
  • id - định danh duy nhất

Event Grid tuân thủ đặc tả CloudEvents 1.0 của Cloud Native Computing Foundation:

JSON
{
    "specversion" : "1.0",
    "type" : "com.yourcompany.order.created",
    "source" : "https://yourcompany.com/orders/",
    "subject" : "O-28964",
    "id" : "A234-1234-1234",
    "time" : "2018-04-05T17:31:00Z",
    "comexampleextension1" : "value",
    "comexampleothervalue" : 5,
    "datacontenttype" : "application/json",
    "data" : {
       "orderId" : "O-28964",
       "URL" : "https://com.yourcompany/orders/O-28964"
    }
}
📏 Giới hạn kích thước

Kích thước tối đa cho một sự kiện là 1 MB. Các sự kiện vượt quá 64 KB bị tính phí theo tăng dần 64 KB.

3. Event Sources (Nguồn sự kiện)

Nguồn sự kiện là nơi sự kiện xảy ra. Mỗi nguồn sự kiện liên quan đến một hoặc nhiều loại sự kiện. Ví dụ:

  • Azure Storage → Blob created events
  • IoT Hub → Device created events
  • Ứng dụng của bạn → Custom events

4. Topics (Chủ đề)

Topic lưu giữ các sự kiện đã được xuất bản đến Event Grid. Có 3 loại topic:

Loại TopicMô Tả
System TopicsTopic tích hợp do các dịch vụ Azure cung cấp. Không thấy trong subscription Azure vì publisher sở hữu topic. Đăng ký bằng cách cung cấp thông tin về tài nguyên muốn nhận sự kiện.
Custom TopicsTopic của ứng dụng và bên thứ ba. Khi bạn tạo hoặc được cấp quyền truy cập vào custom topic, bạn thấy nó trong subscription.
Partner TopicsLoại topic để đăng ký sự kiện từ partner. Sử dụng tính năng Partner Events.

5. Event Subscriptions (Đăng ký sự kiện)

Subscription cho Event Grid biết sự kiện nào trong topic bạn muốn nhận. Khi tạo subscription:

  • Cung cấp endpoint để xử lý sự kiện
  • Có thể lọc theo loại sự kiện hoặc mẫu subject
  • Có thể đặt thời hạn hết hạn

6. Event Handlers (Xử lý sự kiện)

Từ góc độ Event Grid, event handler là nơi sự kiện được gửi đến. Event Grid hỗ trợ nhiều loại handler:

  • HTTP Webhook: Thử lại cho đến khi handler trả về 200 OK
  • Azure Storage Queue: Thử lại cho đến khi Queue service xử lý thành công
  • Azure Functions, Logic Apps, Event Hubs, Service Bus, v.v.

7. Bảo Mật (Security)

Event Grid cung cấp bảo mật cho việc đăng ký topic và xuất bản sự kiện:

  • Khi đăng ký, phải có quyền phù hợp trên Event Grid topic
  • Sử dụng managed identity để xác thực với các service Azure
  • Ví dụ: Để gửi đến Event Hubs, managed identity phải có vai trò Event Hubs Data Sender

Phần 2: Webhooks và Phân Phối Sự Kiện

Phân phối sự kiện đến endpoints qua HTTP

Webhooks là một trong nhiều cách để nhận sự kiện từ Azure Event Grid. Khi một sự kiện mới sẵn sàng, dịch vụ Event Grid sẽ POST HTTP request đến endpoint đã cấu hình với thông tin sự kiện trong request body.

✅ Xác Thực Endpoint (Endpoint Validation)

Event Grid yêu cầu bạn chứng minh quyền sở hữu webhook endpoint trước khi bắt đầu phân phối sự kiện. Điều này ngăn chặn người dùng độc hại flood endpoint của bạn.

Phương thức xác thực 1: Handshake đồng bộ

Event Grid gửi sự kiện Microsoft.EventGrid.SubscriptionValidationEvent với validationCode. Endpoint phải echo lại code này trong response body.

C#
// C# - Xử lý SubscriptionValidationEvent
using Azure.Messaging.EventGrid;
using Azure.Messaging.EventGrid.SystemEvents;

public static async Task<IActionResult> Run(HttpRequest req, ILogger log)
{
    BinaryData events = await BinaryData.FromStreamAsync(req.Body);
    EventGridEvent[] eventGridEvents = EventGridEvent.ParseMany(events);

    foreach (EventGridEvent eventGridEvent in eventGridEvents)
    {
        if (eventGridEvent.TryGetSystemEventData(out object eventData))
        {
            if (eventData is SubscriptionValidationEventData subscriptionValidationEventData)
            {
                log.LogInformation($"Validation code: {subscriptionValidationEventData.ValidationCode}");
                var responseData = new
                {
                    ValidationResponse = subscriptionValidationEventData.ValidationCode
                };
                return new OkObjectResult(responseData);
            }
        }
    }
    return new OkObjectResult(string.Empty);
}

Dữ liệu Validation Event:

JSON
[{
  "id": "2d1781af-3a4c-4d7c-bd0c-e34b19da4e66",
  "topic": "/subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "subject": "",
  "data": {
    "validationCode": "512d38b6-c7b8-40c8-89fe-f46f9e9622b6"
  },
  "eventType": "Microsoft.EventGrid.SubscriptionValidationEvent",
  "eventTime": "2018-01-25T22:12:19.4556811Z",
  "metadataVersion": "1",
  "dataVersion": "1"
}]

Phương thức xác thực 2: CloudEvents v1.0 Abuse Protection

Khi sử dụng CloudEvents schema, Event Grid sử dụng cơ chế bảo vệ lạm dụng (abuse protection) của CloudEvents v1.0 thông qua phương thức HTTP OPTIONS.

Luồng handshake:

  1. Event Grid gửi HTTP OPTIONS request với header WebHook-Request-Origin
  2. Delivery target phản hồi với WebHook-Allowed-Origin và WebHook-Allowed-Rate
HTTP
# Validation Request
WebHook-Request-Origin: eventemitter.example.com

# Validation Response
WebHook-Allowed-Origin: eventemitter.example.com
WebHook-Allowed-Rate: 120  # requests per minute
💡 Xác thực thủ công

Nếu bạn sử dụng dịch vụ API bên thứ ba (như Zapier hoặc IFTTT) và không thể echo lại validation code theo lập trình, bạn có thể xác thực subscription thủ công bằng URL xác thực được gửi trong validationUrl.

Phần 3: Lọc Sự Kiện Event Grid

Lọc theo subject, event type và dữ liệu payload

Khi tạo event subscription, bạn có ba lựa chọn để lọc sự kiện được gửi đến endpoint:

  1. Event types (Loại sự kiện)
  2. Subject begins with or ends with (Subject bắt đầu hoặc kết thúc bằng)
  3. Advanced fields and operators (Trường nâng cao và toán tử)

1️⃣ Lọc theo Loại Sự Kiện

Theo mặc định, tất cả loại sự kiện cho nguồn sự kiện được gửi đến endpoint. Bạn có thể chỉ gửi một số loại nhất định.

JSON
{
  "filter": {
    "includedEventTypes": [
      "Microsoft.Resources.ResourceWriteFailure",
      "Microsoft.Resources.ResourceWriteSuccess"
    ]
  }
}

Ví dụ với ARM template - lọc blob events:

JSON
{
  "type": "Microsoft.EventGrid/systemTopics/eventSubscriptions",
  "properties": {
    "destination": {
      "properties": { "endpointUrl": "[parameters('endpoint')]" },
      "endpointType": "WebHook"
    },
    "filter": {
      "includedEventTypes": [
        "Microsoft.Storage.BlobCreated",
        "Microsoft.Storage.BlobDeleted"
      ]
    }
  }
}

2️⃣ Lọc theo Subject (Chủ đề)

Để lọc đơn giản theo subject, chỉ định giá trị bắt đầu hoặc kết thúc cho subject.

Ví dụ với Blob Storage events:

Subject của Blob storage events có định dạng:

Code
/blobServices/default/containers/<containername>/blobs/<blobname>
Mục tiêuFilter
Tất cả sự kiện trong storage accountĐể trống subject filter
Events từ blobs trong container có prefixsubjectBeginsWith: /blobServices/default/containers/containerprefix
Events từ blobs trong container cụ thểsubjectBeginsWith: /blobServices/default/containers/containername/
Events từ blobs có suffixsubjectEndsWith: .txt hoặc .log

Cách chia nhỏ: Nếu subject là /A/B/C, subscriber lọc theo /A nhận tất cả events với subject như /A/B/C hoặc /A/D/E. Subscriber lọc theo /A/B nhận ít hơn.

3️⃣ Lọc Nâng Cao (Advanced Filtering)

Để lọc theo giá trị trong các trường data và chỉ định toán tử so sánh, sử dụng tùy chọn lọc nâng cao. Chỉ định:

  • operator type: Loại so sánh
  • key: Trường trong event data để lọc (number, boolean, string, hoặc array)
  • values: Giá trị để so sánh với key

Các toán tử cho số (Numbers):

Toán tửMô tảVí dụ JSON
NumberInKey bằng một trong các giá trị filter"values": [5, 1]
NumberNotInKey không bằng bất kỳ giá trị filter nào"values": [41, 0]
NumberLessThanKey nhỏ hơn giá trị filter"value": 100
NumberGreaterThanKey lớn hơn giá trị filter"value": 20
NumberLessThanOrEqualsKey nhỏ hơn hoặc bằng giá trị filter"value": 100
NumberGreaterThanOrEqualsKey lớn hơn hoặc bằng giá trị filter"value": 30
NumberInRangeKey nằm trong một trong các khoảng"values": [[3.14, 999.95], [3000, 4000]]
NumberNotInRangeKey không nằm trong bất kỳ khoảng nào"values": [[3000, 4000]]

Toán tử cho boolean:

Toán tửMô tả
BoolEqualsKey bằng giá trị boolean được chỉ định

Các toán tử cho string:

Toán tửMô tả
StringContainsKey chứa bất kỳ giá trị filter nào (substring)
StringNotContainsKey không chứa bất kỳ giá trị filter nào
StringBeginsWithKey bắt đầu bằng bất kỳ giá trị filter nào
StringNotBeginsWithKey không bắt đầu bằng bất kỳ giá trị filter nào
StringEndsWithKey kết thúc bằng một trong các giá trị filter
StringNotEndsWithKey không kết thúc bằng bất kỳ giá trị filter nào
StringInKey khớp chính xác với một trong các giá trị filter
StringNotInKey không khớp với bất kỳ giá trị filter nào

Toán tử đặc biệt:

Toán tửMô tả
IsNullOrUndefinedKey là NULL hoặc undefined
IsNotNullKey không phải NULL hoặc undefined

Ví dụ lọc nâng cao:

JSON
// NumberIn filter
"advancedFilters": [{
    "operatorType": "NumberIn",
    "key": "data.counter",
    "values": [5, 1]
}]

// StringContains filter
"advancedFilters": [{
    "operatorType": "StringContains",
    "key": "data.key1",
    "values": ["microsoft", "azure"]
}]

// StringEndsWith filter
"advancedFilters": [{
    "operatorType": "StringEndsWith",
    "key": "data.key1",
    "values": ["jpg", "jpeg", "png"]
}]

// BoolEquals filter
"advancedFilters": [{
    "operatorType": "BoolEquals",
    "key": "data.isEnabled",
    "value": true
}]

Logic AND và OR:

💡 Logic Filters
  • Một filter với nhiều giá trị: Thực hiện phép OR (key phải là một trong các giá trị)
  • Nhiều filter khác nhau: Thực hiện phép AND (mỗi điều kiện filter phải được thỏa mãn)

Giới hạn lọc nâng cao:

  • 25 advanced filters và 25 filter values trên tất cả các filter mỗi Event Grid subscription
  • 512 ký tự mỗi giá trị string
  • Key có ký tự . (dấu chấm) không được hỗ trợ escape

☁️ Lọc với CloudEvents Schema

Với CloudEvents schema, sử dụng các giá trị sau cho key: id, source, type, dataschema, hoặc event data (như data.key1).

Bạn cũng có thể sử dụng extension context attributes trong CloudEvents 1.0:

JSON
{
    "specversion" : "1.0",
    "type" : "com.example.someevent",
    "source" : "/mycontext",
    "comexampleextension1" : "value",
    "comexampleothervalue" : 5,
    "data" : { "appinfoA" : "abc", "appinfoB" : 123 }
}

// Filter theo extension attribute
"advancedFilters": [{
    "operatorType": "StringBeginsWith",
    "key": "comexampleothervalue",
    "values": ["5", "1"]
}]

☁️ CloudEvents (Được Khuyến Nghị)

CloudEvents là định dạng sự kiện được khuyến nghị, cung cấp định nghĩa tiêu chuẩn và không phụ thuộc giao thức. Event Grid đầu tư vào các tính năng liên quan đến CloudEvents JSON format.

Bảng chuyển đổi schema được hỗ trợ:

Input SchemaOutput Schema
CloudEvents formatCloudEvents format
Event Grid formatCloudEvents format
Event Grid formatEvent Grid format
⚠️ Lưu ý quan trọng

Output schema kiểu Event Grid không thể sử dụng khi CloudEvents là input schema, vì CloudEvents hỗ trợ extension attributes mà Event Grid schema không hỗ trợ.

Giới hạn kích thước:

  • Mảng sự kiện khi POST đến topic: tối đa 1 MB
  • Mỗi sự kiện trong mảng: tối đa 1 MB
  • Các thao tác tính phí theo tăng dần 64 KB (event 130 KB bị tính như 3 event)
  • Event Grid gửi đến subscriber trong mảng chỉ có một sự kiện

Phần 5: Azure Event Hubs — Streaming Thời Gian Thực

Nền tảng streaming dữ liệu lớn thời gian thực

Azure Event Hubs là dịch vụ phát trực tuyến dữ liệu gốc trên đám mây, có thể phát trực tuyến hàng triệu sự kiện mỗi giây với độ trễ thấp, từ bất kỳ nguồn nào đến bất kỳ đích nào. Event Hubs tương thích với Apache Kafka, cho phép chạy các workload Kafka hiện có mà không cần thay đổi code.

🔑 Các Khả Năng Chính

1. Apache Kafka trên Azure Event Hubs

Event Hubs là engine phát trực tuyến sự kiện đa giao thức, hỗ trợ gốc AMQP, Apache Kafka và HTTPS. Bạn có thể mang Kafka workloads sang Event Hubs mà không thay đổi code.

2. Schema Registry

Azure Schema Registry trong Event Hubs cung cấp kho lưu trữ tập trung để quản lý schema của ứng dụng phát trực tuyến sự kiện. Schema Registry đi kèm miễn phí với mỗi Event Hubs namespace.

3. Xử lý Stream Thời Gian Thực với Stream Analytics

Event Hubs tích hợp với Azure Stream Analytics để kích hoạt xử lý stream thời gian thực. Với trình chỉnh sửa no-code trong portal, bạn có thể phát triển Stream Analytics job bằng drag-and-drop.

Phần 6: Partitions trong Event Hubs

Phân vùng song song để tăng throughput xử lý

📐 Partition là gì?

Partition là cơ chế tổ chức dữ liệu cho phép publish và consume song song. Partition có thể được coi như commit log. Partitions lưu giữ event data chứa:

  • Body của sự kiện
  • Property bag do người dùng định nghĩa
  • Metadata như offset trong partition, sequence number trong stream
  • Service-side timestamp khi được chấp nhận
P0
Partition 0 — Commit Log (đọc trái → phải) offset=0 [Event 1] → offset=1 [Event 2] → offset=2 [Event 3] → offset=3 [Event 4] → offset=4 [Event 5]
← Cũ hơn                                                          Mới hơn →
E
Epoch (Exclusive) Consumers — mỗi partition 1 owner P0 → Consumer A  |  P1 → Consumer B  |  P2 → Consumer A  |  P3 → Consumer C
N
Non-Epoch Consumers — nhiều readers cùng partition P0 → Consumer 1, Consumer 2, Consumer 3 (fan-out, tối đa 5) — không tăng throughput

✅ Lợi Ích Của Partitions

  • Tăng throughput: Nhiều partition song song → nhân throughput IO
  • Xử lý song song: Nhiều tiến trình consumer xử lý song song
  • Scalability: Mỗi partition là owner độc lập

Hướng dẫn throughput mỗi partition:

TầngIngressEgress
Standard~1 MB/s~2 MB/s
Premium và Dedicated~1-2 MB/s~2-5 MB/s

🔢 Số Lượng Partitions

  • Được chỉ định khi tạo event hub
  • Đối với tầng Standard: không thể thay đổi sau khi tạo
  • Đối với tầng Premium/Dedicated: có thể tăng (không giảm) sau khi tạo
  • Số partition = số consumer song song tối đa trên mỗi consumer group
💡 Không ảnh hưởng đến giá

Số lượng partition không ảnh hưởng đến giá. Giá phụ thuộc vào throughput units (Standard), processing units (Premium) hoặc capacity units (Dedicated).

🗺️ Ánh Xạ Events đến Partitions

Bạn có thể sử dụng partition key để ánh xạ event data đến partitions cụ thể. Partition key được xử lý qua hàm hashing tĩnh.

  • Nếu không chỉ định partition key: sử dụng round-robin assignment
  • Partition key giúp giữ các sự kiện liên quan cùng partition và theo thứ tự đến
  • Ví dụ key tốt: ID thiết bị hoặc người dùng, địa lý
⚠️ Không nên gửi trực tiếp đến partition

Không nên gửi sự kiện trực tiếp đến partition cụ thể, đặc biệt khi high availability quan trọng. Điều này làm giảm tính sẵn sàng của event hub xuống cấp partition.

👥 Consumer Groups và Offsets

Consumer Groups:

Consumer group là view độc lập của event stream. Mỗi consumer group có thể đọc cùng event hub đồng thời, mỗi group theo dõi vị trí của riêng mình.

  • Mỗi event hub có consumer group mặc định $Default
  • Tạo consumer group riêng cho mỗi ứng dụng (analytics, archival, alerting)
  • Một active reader mỗi partition trong một consumer group

Offsets:

Offset là vị trí của sự kiện trong partition - như con trỏ. Consumers sử dụng offset để chỉ định nơi bắt đầu đọc. Có thể bắt đầu từ:

  • Giá trị offset cụ thể
  • Timestamp
  • Đầu hoặc cuối stream

Checkpointing:

Checkpointing là khi consumer lưu offset hiện tại. Điều này cho phép:

  • ✅ Tiếp tục từ lần checkpoint cuối cùng sau khi ngắt kết nối
  • ✅ Failover: Consumer instance mới tiếp quản từ nơi consumer cũ bỏ dở
  • ✅ Replay: Xử lý lại sự kiện lịch sử bằng offset cũ hơn
⚠️ Checkpoint Store Best Practices
  • Sử dụng container riêng cho mỗi consumer group
  • Không sử dụng storage account cho mục đích khác
  • Tắt: Hierarchical namespace, Blob soft delete, Versioning
  • Tạo storage account trong cùng region với ứng dụng

Phần 7: Event Hubs Capture

Lưu trữ tự động sự kiện vào Storage hoặc Data Lake

Tính năng Azure Event Hubs Capture tự động chụp dữ liệu streaming qua Event Hubs vào tài khoản Azure Blob Storage hoặc Azure Data Lake Storage. Capture không yêu cầu chi phí quản trị để chạy và tự động scale với Event Hubs.

1
Producers (Nguồn)Gửi events đến Event Hub qua AMQP, Kafka, hoặc HTTPS
2
Event Hub — Partitions 0–3Lưu events theo thứ tự trên từng partition. Cả Capture và consumers bình thường đọc song song.
3a
Capture → Azure Blob Storage / Data Lake StorageĐịnh dạng Avro. Kích hoạt theo cửa sổ: min(size_threshold, time_threshold) → "First Wins"
Đặt tên: {Namespace}/{EventHub}/{PartitionId}/{Year}/{Month}/{Day}/{Hour}/{Minute}/{Second}.avro
3b
Stream Analytics / Apache Spark (consumers bình thường)Đọc và xử lý real-time song song với Capture — không ảnh hưởng lẫn nhau

⚙️ Cách Event Hubs Capture Hoạt Động

Event Hubs phục vụ như một durable buffer giữ lâu dài cho telemetry ingress, tương tự distributed log. Capture cho phép:

  • Xử lý real-time và batch-based pipelines trên cùng một stream
  • Giúp xây dựng giải pháp phát triển theo nhu cầu

Điểm quan trọng cần lưu ý:

  • Tài khoản storage đích phải cùng subscription với event hub (khi không dùng managed identity)
  • Event Hubs không hỗ trợ capture đến premium Azure Storage accounts
  • Dữ liệu được ghi theo định dạng Apache Avro - compact, binary, có rich data structures với inline schema

⏱️ Capture Windowing (Cửa sổ Capture)

Để kiểm soát việc capture, sử dụng cấu hình cửa sổ kết hợp kích thước tối thiểu và thời gian:

💡 Chính sách "First Wins"

Điều kiện đầu tiên được đáp ứng - kích thước hoặc thời gian - sẽ kích hoạt capture. Ví dụ: cửa sổ 15 phút, 100MB và gửi 1 MB/giây → cửa sổ kích thước kích hoạt trước cửa sổ thời gian.

Cấu trúc đặt tên file capture:

JSON
{Namespace}/{EventHub}/{PartitionId}/{Year}/{Month}/{Day}/{Hour}/{Minute}/{Second}

# Ví dụ:
https://mystorageaccount.blob.core.windows.net/mycontainer/mynamespace/myeventhub/0/2017/12/08/03/03/17.avro

💰 Thanh Toán Capture

  • Premium tier: Tính năng Capture được bao gồm
  • Standard tier: Azure tính phí hàng tháng dựa trên số throughput units
  • Capture không tiêu thụ egress quota vì Azure tính phí riêng

Phần 8: Xử Lý Theo Tỷ Lệ với Event Hubs

📋 Ví Dụ Tình Huống Thực Tế

Công ty bảo mật nhà giám sát 100,000 ngôi nhà. Mỗi phút, nhận dữ liệu từ các cảm biến (detector chuyển động, cảm biến cửa/cửa sổ, v.v.):

  • Mỗi cảm biến đẩy dữ liệu đến một event hub
  • Event hub được cấu hình với 16 partitions
  • Consumer cần đọc, tổng hợp và lưu vào storage blob

🏗️ Yêu Cầu Consumer Phân Tán

  1. Scale: Tạo nhiều consumer, mỗi consumer sở hữu một số partition của Event Hubs
  2. Load balance: Tăng/giảm consumer động. Khi thêm consumer, pool tự cân bằng lại số partition
  3. Seamless resume on failures: Nếu consumer A thất bại, consumer khác tiếp tục từ checkpoint cuối cùng của consumer A
  4. Consume events: Code để tiêu thụ sự kiện và làm gì đó hữu ích

✅ Khuyến nghị Production

Đối với hầu hết tình huống production, khuyến nghị sử dụng event processor client để đọc và xử lý sự kiện. Processor client cung cấp trải nghiệm mạnh mẽ cho việc xử lý sự kiện trên tất cả partition theo cách hiệu suất và chịu lỗi.

🏷️ Partition Ownership (Quyền sở hữu Partition)

Instance event processor thường sở hữu và xử lý sự kiện từ một hoặc nhiều partition. Quyền sở hữu partition được phân phối đều giữa tất cả instance event processor đang hoạt động.

Bản ghi quyền sở hữu trong checkpoint store:

NamespaceEventHubConsumerGroupOwnerPartitionIdLastModified
mynamespace.servicebus.windows.netmyeventhubmyconsumergroup3be3f9d3-...02020-01-15T01:22:15
mynamespace.servicebus.windows.netmyeventhubmyconsumergroupf5cc5176-...12020-01-15T01:22:17

Khi processor thất bại:

  1. Các instance khác phát hiện qua last modified time
  2. Thử giành quyền sở hữu partition của instance không hoạt động
  3. Checkpoint store đảm bảo chỉ một instance thành công
  4. Tại bất kỳ thời điểm nào, tối đa một processor nhận sự kiện từ partition

✅ Checkpointing Chi Tiết

Checkpointing là quá trình event processor đánh dấu hoặc commit vị trí của sự kiện cuối cùng được xử lý thành công trong partition.

  • Xảy ra trên cơ sở mỗi partition trong mỗi consumer group
  • Khi processor ngắt kết nối, instance khác tiếp tục từ checkpoint được commit bởi processor cuối cùng
  • Có thể quay về dữ liệu cũ hơn bằng cách chỉ định offset thấp hơn
⚠️ Cân nhắc về tần suất checkpointing

Cập nhật sau mỗi sự kiện được xử lý thành công có thể ảnh hưởng đến hiệu suất và chi phí vì nó kích hoạt write operation đến checkpoint store. Checkpointing mỗi sự kiện đơn lẻ cũng cho thấy pattern queued messaging - Service Bus queue có thể là lựa chọn tốt hơn.

🧵 An Toàn Thread và Instances

Theo mặc định, function xử lý sự kiện được gọi tuần tự cho một partition cụ thể. Sự kiện tiếp theo và các lệnh gọi từ cùng partition xếp hàng phía sau. Sự kiện từ các partition khác nhau có thể được xử lý đồng thời.

💡 Lưu ý về Shared State

Bất kỳ shared state nào được truy cập qua các partition phải được đồng bộ hóa.

📋 Tóm Tắt Quan Trọng Cho Kỳ Thi

✅ Điểm cần nhớ - Event Grid
  • Event Grid hỗ trợ PUSH và PULL delivery
  • Hỗ trợ CloudEvents 1.0 specification (được khuyến nghị)
  • 3 loại topic: System, Custom, Partner
  • Xác thực webhook qua SubscriptionValidationEvent hoặc HTTP OPTIONS (CloudEvents)
  • 3 cách lọc: Event type, Subject, Advanced
  • Advanced filter: OR trong một filter, AND giữa các filter khác nhau
  • Giới hạn: 25 advanced filters, 512 chars/string, 1MB/event
✅ Điểm cần nhớ - Event Hubs
  • Event Hubs tương thích Apache Kafka, AMQP, HTTPS
  • Partition = commit log append-only
  • Consumer Group = view độc lập, nhiều app đọc cùng data
  • Offset = vị trí trong partition; Checkpointing = lưu offset
  • Capture: Avro format, windowing (size hoặc time first wins)
  • EventProcessorClient: tự động load balancing và checkpointing
  • Standard: không thể thay đổi partition count; Premium/Dedicated: có thể tăng
  • Retention: Standard 7 ngày, Premium/Dedicated 90 ngày
