Module 1: Azure Key Vault

Lưu trữ bảo mật và truy cập secrets, keys và certificates

🔷 1.1 Tổng Quan Azure Key Vault

Azure Key Vault hỗ trợ hai loại containers:

  • Vaults — Lưu trữ software và HSM-backed keys, secrets và certificates
  • Managed HSM pools — Chỉ hỗ trợ HSM-backed keys

📜 Certificate Management

Auto-renew from CAs

🔹 Service Tiers

TierBảo vệ KeysMô tả

PremiumHSM-protected keyDùng Hardware Security Module (HSM). Tuân thủ FIPS 140-2 Level 2/3. Cho workloads yêu cầu compliance cao.

🔹 Lợi Ích Chính

📌 Authentication Flow

Authentication vào Key Vault sử dụng Microsoft Entra ID. Caller (user hoặc app) cần xác thực với Entra ID trước, sau đó dùng token để truy cập Key Vault.

🔹 Best Practices Bảo Mật

  • Dùng Managed Identity — Không cần lưu credentials trong code
  • Tách biệt Key Vaults — Mỗi môi trường (Dev/Staging/Prod) một vault riêng
  • Bật logging — Monitor mọi truy cập vào vault
  • Bật soft-delete và purge protection — Tránh xóa nhầm secrets
  • Rotate secrets định kỳ — Tự động hoặc thủ công
  • Hạn chế network access — Dùng Private Endpoints hoặc service endpoints

🔹 Lưu Trữ và Lấy Secret — Code Example (.NET)

C#
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

// Tạo client với DefaultAzureCredential (tự động dùng Managed Identity khi deploy)
var client = new SecretClient(
    vaultUri: new Uri("https://myvault.vault.azure.net/"),
    credential: new DefaultAzureCredential()
);

// Lưu secret
await client.SetSecretAsync("my-secret", "secret-value");

// Lấy secret
KeyVaultSecret secret = await client.GetSecretAsync("my-secret");
string secretValue = secret.Value;

// Lấy version cụ thể
KeyVaultSecret secretV2 = await client.GetSecretAsync("my-secret", "version-id");
CLI
# Azure CLI
# Lưu secret
az keyvault secret set --vault-name MyVault --name MySecret --value "MySecretValue"

# Lấy secret
az keyvault secret show --vault-name MyVault --name MySecret

# List secrets
az keyvault secret list --vault-name MyVault

🔹 Key Vault URI Format

CLI
# Secret URI format:
https://{vault-name}.vault.azure.net/secrets/{secret-name}/{version}

# Ví dụ:
https://myvault.vault.azure.net/secrets/my-db-password/abc123def456

Module 2: Managed Identities

Danh tính được quản lý tự động — không cần credentials trong code

🔷 2.1 Managed Identities — Khái Niệm

Managed Identities giải quyết thách thức lớn của developer: quản lý credentials, secrets và certificates để kết nối giữa các dịch vụ. Managed Identities cung cấp danh tính tự động quản lý trong Microsoft Entra ID cho applications.

✅ Lợi Ích Cốt Lõi

Applications có thể dùng Managed Identity để lấy Microsoft Entra tokens mà không cần quản lý credentials nào. Nội bộ, managed identities là service principals đặc biệt — chỉ dùng được với Azure resources.

🔹 Hai Loại Managed Identity

Đặc điểm ⚙️ System-Assigned (Gắn với 1 resource) 👥 User-Assigned (Resource độc lập)
Tạo Tự động khi enable trên Azure resource (VM, App, Func...) Tạo thủ công như một Azure resource riêng biệt
Lifecycle Gắn với resource parent — tự xóa khi resource bị xóa Quản lý RIÊNG — phải xóa tường minh
Sharing ❌ Không thể share
(1 resource = 1 identity)		 ✅ Gán cho nhiều resources cùng lúc
Dùng khi Workload chạy trên 1 resource duy nhất Workload trên nhiều resources, cần shared identity, preauthorize trước khi tạo resource

🔹 So Sánh Chi Tiết

Thuộc TínhSystem-AssignedUser-Assigned
Cách tạoTạo cùng với Azure resourceTạo như standalone Azure resource
Vòng đờiGắn chặt với resource cha. Xóa resource → identity tự xóaĐộc lập. Phải xóa tường minh.
Chia sẻKhông thể chia sẻ — chỉ liên kết với 1 resourceCó thể chia sẻ với nhiều resources
Use case1 VM cần truy cập Key VaultNhiều VMs cần cùng 1 identity, workload recycling thường xuyên

🔷 2.2 Cấu Hình Managed Identity

🔹 System-Assigned — Azure CLI

CLI
# Tạo VM với system-assigned identity
az vm create \
    --resource-group myResourceGroup \
    --name myVM \
    --image win2016datacenter \
    --generate-ssh-keys \
    --assign-identity \          # Bật system-assigned identity
    --role contributor \
    --scope mySubscription \
    --admin-username azureuser \
    --admin-password myPassword12

# Bật system-assigned identity trên VM đã có
az vm identity assign \
    -g myResourceGroup \
    -n myVM

🔹 User-Assigned — Azure CLI

CLI
# Bước 1: Tạo user-assigned identity
az identity create \
    -g myResourceGroup \
    -n myUserAssignedIdentity

# Bước 2a: Tạo VM mới với user-assigned identity
az vm create \
    --resource-group myResourceGroup \
    --name myVM \
    --image Ubuntu2204 \
    --admin-username azureuser \
    --admin-password myPassword12 \
    --assign-identity myUserAssignedIdentity \
    --role contributor \
    --scope /subscriptions/xxx

# Bước 2b: Gán identity cho VM đã có
az vm identity assign \
    -g myResourceGroup \
    -n myVM \
    --identities myUserAssignedIdentity

🔷 2.3 Lấy Access Token — DefaultAzureCredential

Cách khuyến nghị để lấy access token là dùng DefaultAzureCredential từ Azure Identity library. Nó tự động thử nhiều phương thức xác thực theo thứ tự.

🔹 Thứ Tự DefaultAzureCredential

Thử lần lượt theo thứ tự — dừng khi tìm được credentials hợp lệ:

1
🔧 Environment VariablesAZURE_CLIENT_ID, AZURE_CLIENT_SECRET, AZURE_TENANT_ID
2
🖥️ Workload IdentityKubernetes workload identity
3
☁️ Managed IdentityKhi deploy trên Azure host có Managed Identity
4
💻 Visual StudioLocal development credentials
5
🔧 Azure CLIaz login
6
🔷 Azure PowerShellConnect-AzAccount
7
🌐 Interactive BrowserDisabled by default
✅ Lợi Ích DefaultAzureCredential

Cùng một code hoạt động trong development (dùng Azure CLI credentials) và production (dùng Managed Identity) — không cần thay đổi code.

🔹 Code Examples (.NET)

CLI
# Cài đặt package
dotnet add package Azure.Identity
C#
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;
using Azure.Storage.Blobs;
using Azure.Messaging.EventHubs;

// ── Authenticate với Key Vault ──
var secretClient = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential()
);

// ── Với User-Assigned Managed Identity cụ thể ──
string clientId = "<managed-identity-client-id>";
var credential = new DefaultAzureCredential(
    new DefaultAzureCredentialOptions { ManagedIdentityClientId = clientId }
);
var blobClient = new BlobClient(
    new Uri("https://myaccount.blob.core.windows.net/mycontainer/myblob"),
    credential
);

// ── Custom Chain (Managed Identity → Azure CLI fallback) ──
var chainedCredential = new ChainedTokenCredential(
    new ManagedIdentityCredential(),
    new AzureCliCredential()
);
var eventHubClient = new EventHubProducerClient(
    "myeventhub.eventhubs.windows.net",
    "myhubpath",
    chainedCredential
);

Module 3: Azure App Configuration

Quản lý cài đặt ứng dụng và feature flags tập trung

🔷 3.1 Tổng Quan App Configuration

Azure App Configuration cung cấp dịch vụ quản lý tập trung application settings và feature flags. Đặc biệt hữu ích với microservices và distributed applications.

🔹 App Configuration vs Key Vault

Dùng cho ⚙️ App Configuration 🔐 Key Vault
✅ Phù hợp Non-sensitive config
Feature flags
Settings shared across services
Dynamic config updates
Label-based environments		 Secrets & passwords
Encryption keys
Certificates
Connection strings (sensitive)
🔒 HSM-backed security
❌ Không phù hợp Secrets (passwords, keys) Feature flags, non-secret config

💡 Thường dùng CẢ HAI: App Config lưu settings, Key Vault lưu secrets. App Config có thể reference đến Key Vault secrets.

🔹 Lợi Ích

  • Dịch vụ fully managed, thiết lập trong vài phút
  • Flexible key representations và mappings
  • Labels để phân biệt environments (Dev/Staging/Prod)
  • Point-in-time replay — khôi phục settings tại thời điểm cụ thể
  • Feature flag UI chuyên dụng
  • Thay đổi settings không cần redeploy/restart
  • Mã hóa at rest và in transit

🔹 Kết Nối Theo Ngôn Ngữ

🔷 3.2 Keys, Values và Labels

🔹 Cấu Trúc Keys

Keys là tên của key-value pairs. Thường tổ chức theo hierarchical namespace dùng dấu phân cách / hoặc :.

CLI
# Ví dụ hierarchical keys
AppName:Service1:ApiEndpoint
AppName:Service2:ApiEndpoint
AppName:Database:ConnectionString

# Spring Cloud format
{application}/{profile}/key
⚠️ Quy Tắc Đặt Tên Keys
  • Keys là case-sensitive — app1 và App1 là hai keys khác nhau
  • Ký tự reserved (không được dùng): *, ,, \ — phải escape: \{char}
  • Giới hạn 10,000 ký tự cho key-value pair (key + value + attributes)

🔹 Labels — Phân Biệt Environments

Labels là attribute tùy chọn để phân biệt key-values có cùng key:

CLI
# Cùng key, khác label cho mỗi environment:
Key = AppName:DbEndpoint & Label = Test
Key = AppName:DbEndpoint & Label = Staging
Key = AppName:DbEndpoint & Label = Production

# Query specific version:
Key = AppName:DbEndpoint & Label = 2024-03-01-release
💡 App Config KHÔNG tự động version

Dùng labels để tạo versions: đặt app version number hoặc Git commit ID làm label để track settings theo từng software build.

🔹 Values

  • Values là Unicode strings
  • Content type tùy chọn (ví dụ: application/json)
  • Data được mã hóa at rest và in transit
  • KHÔNG lưu application secrets trong App Configuration — dùng Key Vault

🔷 3.3 Feature Flags (Cờ Tính Năng)

Feature flags cho phép bật/tắt tính năng ứng dụng runtime mà không cần redeploy. App Configuration cung cấp Feature Manager tích hợp.

🔹 Khái Niệm Feature Management

🚦 Feature Flag Structure
  • name: "BetaFeature"
  • state: enabled | disabled
  • Percentage filter: Value: 20 (20% users)
  • TimeWindow filter: Start/End datetime
  • Targeting filter: specific users/groups
🎯 Scenarios
  • Canary release — 10% users thấy tính năng mới
  • A/B testing — nhóm A vs nhóm B
  • Beta testing — chỉ specific users
  • Kill switch — tắt ngay khi có lỗi, không cần redeploy

🔹 Feature Manager trong ASP.NET Core

CLI
# Cài packages
dotnet add package Microsoft.FeatureManagement.AspNetCore
C#
// Program.cs - Đăng ký Feature Management
builder.Services.AddFeatureManagement();

// Kết nối App Configuration
builder.Configuration.AddAzureAppConfiguration(options => {
    options.Connect(connectionString)
           .UseFeatureFlags();
});

// Sử dụng trong Controller
public class HomeController : Controller
{
    private readonly IFeatureManager _featureManager;

    public HomeController(IFeatureManager featureManager)
        => _featureManager = featureManager;

    public async Task<IActionResult> Index()
    {
        if (await _featureManager.IsEnabledAsync("BetaFeature"))
        {
            return View("BetaHome");
        }
        return View("Home");
    }
}

// Attribute-based trong ASP.NET Core
[FeatureGate("PremiumFeature")]
public IActionResult PremiumContent()
{
    return View();
}

🔹 Lưu Feature Flag trong App Configuration

JSON
// Format của feature flag trong App Config:
{
  "id": "BetaFeature",
  "description": "Tính năng Beta cho testing",
  "enabled": true,
  "conditions": {
    "client_filters": [
      {
        "name": "Microsoft.Percentage",
        "parameters": {
          "Value": 20
        }
      }
    ]
  }
}

🔷 3.4 Bảo Mật App Configuration

  • Dùng Managed Identity để kết nối — không cần connection string
  • App Config tích hợp với Key Vault references — tham chiếu đến secrets trong Key Vault mà không copy secret sang App Config
  • Hỗ trợ Private Link để truy cập qua private network
  • RBAC để kiểm soát ai được đọc/ghi configuration
C#
// Key Vault reference trong App Configuration
// App Config lưu reference đến KV secret, không lưu giá trị thật
{
  "key": "Database:ConnectionString",
  "value": "{\"uri\":\"https://myvault.vault.azure.net/secrets/db-connection\"}",
  "contentType": "application/vnd.microsoft.appconfig.keyvaultref+json;charset=utf-8"
}

Tóm Tắt — Điểm Quan Trọng Cho Kỳ Thi

Tắt — Điểm Quan Trọng Cho Kỳ Thi
🔐 BẢO MẬT AZURE — TÓM TẮT
🔑 Key Vault
  • 2 containers: Vaults (keys+secrets+certs) và Managed HSM pools (chỉ keys)
  • 2 tiers: Standard (software key) và Premium (HSM-protected)
  • 2 auth models: RBAC (khuyến nghị) và Access Policy (legacy)
  • URI: https://{vault}.vault.azure.net/secrets/{name}/{version}
  • Bật soft-delete + purge protection để tránh mất data
🆔 Managed Identities
  • System-assigned: gắn với 1 resource, tự xóa khi resource xóa
  • User-assigned: resource độc lập, có thể share với nhiều resources
  • DefaultAzureCredential: tự động dùng Managed Identity khi deploy
  • ChainedTokenCredential: custom chain (MI → CLI → etc.)
⚙️ App Configuration
  • Keys: case-sensitive, hierarchical (dùng / hoặc :)
  • Labels: phân biệt environments với cùng một key
  • KHÔNG thay thế Key Vault — lưu settings (không phải secrets)
  • Feature flags: bật/tắt tính năng runtime không cần redeploy
  • Key Vault references: tham chiếu đến KV secrets
  • Khi nào dùng Key Vault: secrets, encryption keys, certificates
  • Khi nào dùng App Configuration: non-sensitive settings, feature flags
  • Khi nào dùng System-assigned identity: 1 resource, lifecycle gắn cùng
  • Khi nào dùng User-assigned identity: nhiều resources, lifecycle độc lập
  • DefaultAzureCredential ưu tiên: Environment → Managed Identity → VS → CLI → PS
