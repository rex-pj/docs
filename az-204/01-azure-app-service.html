Module 1: Khám Phá Azure App Service

Nền tảng PaaS được quản lý hoàn toàn

1.1 Azure App Service Là Gì?

Azure App Service là nền tảng PaaS được quản lý hoàn toàn — không cần quản lý hạ tầng, chỉ tập trung viết code. Hỗ trợ .NET, Java, Node.js, Python, PHP trên Windows hoặc Linux, bao gồm custom containers.

1.2 App Service Plans

Mỗi app luôn chạy trong một App Service Plan — đơn vị xác định tài nguyên tính toán, OS, vùng địa lý, số instances, và pricing tier. Nhiều apps có thể dùng chung một plan và chia sẻ tài nguyên.

📌 App Service Plan = Scale Unit
  • 5 instances trong plan → tất cả apps trong plan chạy trên cả 5 instances
  • Deployment slots, WebJobs, diagnostic logs đều dùng CPU/RAM của plan
  • Autoscale áp dụng cho tất cả apps trong plan cùng lúc
  • Nên tách app sang plan riêng khi: tiêu tốn nhiều tài nguyên, cần scale độc lập, cần region khác

1.4 Xác Thực & Phân Quyền (Built-in Auth)

Auth middleware tích hợp sẵn chạy trên cùng VM với app (Linux: trong container riêng). Mọi HTTP request đều qua middleware trước khi đến app code — không cần thay đổi app code.

Identity Providers: Entra ID · Facebook · Google · X · GitHub · Apple · OpenID Connect bất kỳ

🌐 Server-Directed Flow
Dành cho: Browser apps (không có Provider SDK)
1
Browser gửi request tới /.auth/login/<provider>
2
Provider redirect về /.auth/login/<p>/callback
3
App Service thêm auth cookie vào response
4
Browser tự gắn cookie vào các request tiếp theo
📱 Client-Directed Flow
Dành cho: REST API, SPA, Mobile (có Provider SDK)
1
App code dùng Provider SDK để đăng nhập
2
Nhận auth token từ provider
3
POST token lên /.auth/login/<provider>
4
App Service trả X-ZUMO-AUTH token
⚠️ Lưu Ý Phân Quyền

Cấu hình "Yêu cầu xác thực" áp dụng cho tất cả calls → không phù hợp với apps có trang chủ công khai (SPAs). Dùng "Cho phép request chưa xác thực" và tự xử lý trong app code nếu cần linh hoạt.

Module 2: Cấu Hình Web App Settings

App settings, connection strings, diagnostics

2.1 Application Settings

App settings được truyền vào app như environment variables. App Service restart app khi thêm, xóa hoặc sửa settings. Với ASP.NET/Core: App Settings ghi đè Web.config / appsettings.json.

📌 Linux/Container — Nested Keys

Nested JSON key ApplicationInsights:InstrumentationKey phải viết thành ApplicationInsights__InstrumentationKey — dùng double underscore __ thay :

JSON
[
  { "name": "MY_KEY",      "value": "my-value", "slotSetting": false },
  { "name": "DB_CONN",     "value": "Server=...", "slotSetting": true  }
  //                                              ↑ slot-sticky — KHÔNG swap
]
Azure CLI
az webapp config appsettings set \
  --resource-group <group> --name <app> \
  --settings KEY1=value1 KEY2=value2

3.3 Automatic Scaling (HTTP-based)

Bật ở cấp App Service Plan, cấu hình range instances per app. Platform tự monitor HTTP load và scale — không cần định nghĩa rules.

  • Không muốn cấu hình metric rules phức tạp
  • Các apps trong cùng Plan cần scale độc lập
  • Muốn giới hạn max instances để tránh overwhelm backend (DB, legacy system)

Module 4: Deployment Slots

Swap không downtime giữa staging và production

4.1 Deployment Slots Là Gì?

Deployment Slots là các live apps với hostname riêng, chạy trong cùng App Service Plan. Nội dung và cấu hình có thể swap giữa các slots kể cả production slot.

📌 Yêu Cầu & URL Format
  • Tier Standard, Premium hoặc Isolated. Standard: tối đa 5 slots
  • URL: http://<sitename>-<slotname>.azurewebsites.net
  • Tổng site name + slot name phải < 59 ký tự
  • Validate trước khi swap — test staging trước khi vào production
  • Zero downtime — slot được warm up trước khi swap
  • Rollback tức thì — swap ngược lại để khôi phục version cũ
  • Auto swap — tự động swap khi không cần preswap validation

4.3 Điều Phối Traffic (Traffic Routing)

A/B Testing: Chỉ định % traffic route tới slot không phải production. Client được route vào slot sẽ "ghim" suốt session qua cookie x-ms-routing-name.

HTML
<!-- User opt-in thử beta -->
<a href="myapp.azurewebsites.net/?x-ms-routing-name=staging">Thử phiên bản beta</a>

<!-- User quay về production -->
<a href="myapp.azurewebsites.net/?x-ms-routing-name=self">Quay về ổn định</a>
💡 Hidden Staging Slot

Giữ routing % = 0% → slot ẩn với public. Team nội bộ vẫn truy cập được bằng query parameter để test.

📋 Điểm Quan Trọng Cho Kỳ Thi

  • App Service Plan = scale unit — tất cả apps trong plan scale cùng nhau
  • Tier Free/Shared không scale out được, chỉ dùng dev/test
  • Deployment Slots yêu cầu tier Standard trở lên (Standard: 5 slots)
  • Swap = swap hostnames — không copy data, không restart app
  • App Settings ghi đè Web.config / appsettings.json trong ASP.NET
  • Linux/Container: nested key dùng __ (double underscore) thay :
  • Built-in auth chạy trong middleware riêng, không cần thay đổi app code
  • Autoscale = thay đổi số instances (scale out) — không thay đổi CPU/RAM
  • Automatic Scaling chỉ khả dụng với Premium V2, Premium V3
  • .NET + PostgreSQL → dùng connection string type Custom (không phải PostgreSQL)
