Module 1: Microsoft Identity Platform

Nền tảng xác thực và phân quyền hiện đại của Microsoft

Tổng quan về Microsoft Identity Platform

Microsoft Identity Platform giúp bạn xây dựng các ứng dụng mà người dùng và khách hàng có thể đăng nhập bằng tài khoản Microsoft hoặc tài khoản mạng xã hội, đồng thời cung cấp quyền truy cập được ủy quyền vào các API của bạn hoặc API của Microsoft như Microsoft Graph.

Các thành phần chính của Microsoft Identity Platform:

  • 🔐 Dịch vụ xác thực tuân thủ OAuth 2.0 và OpenID Connect — cho phép các nhà phát triển xác thực nhiều loại danh tính:
    • Tài khoản cơ quan hoặc trường học (qua Microsoft Entra ID)
    • Tài khoản Microsoft cá nhân (Xbox, Outlook.com)
    • Tài khoản mạng xã hội hoặc địa phương (qua Azure AD B2C)
    • Tài khoản khách hàng (qua Microsoft Entra External ID)
  • 📚 Thư viện mã nguồn mở — Microsoft Authentication Libraries (MSAL) và hỗ trợ các thư viện tuân thủ tiêu chuẩn khác
  • 🌐 Endpoint của Microsoft Identity Platform — đã được chứng nhận OIDC, hoạt động với MSAL hoặc bất kỳ thư viện tuân thủ tiêu chuẩn nào. Triển khai các phạm vi dễ đọc theo tiêu chuẩn ngành
  • 🏢 Cổng quản lý ứng dụng — Trải nghiệm đăng ký và cấu hình trong Azure portal, cùng với các khả năng quản lý Azure khác
  • ⚙️ API cấu hình ứng dụng và PowerShell — Cấu hình ứng dụng theo chương trình thông qua Microsoft Graph API và PowerShell để tự động hóa các tác vụ DevOps
💡 Lợi ích cho nhà phát triển

Microsoft Identity Platform cung cấp tích hợp các đổi mới hiện đại trong không gian danh tính và bảo mật như xác thực không mật khẩu (passwordless), xác thực nâng cao (step-up authentication) và Conditional Access. Các ứng dụng tích hợp với nền tảng này sẽ được hưởng lợi tự động từ các đổi mới đó.

Module 2: Service Principals & App Registration

Đăng ký ứng dụng và quản lý service principals

Đăng ký ứng dụng (App Registration)

Để ủy quyền các chức năng Quản lý Danh tính và Truy cập cho Microsoft Entra ID, ứng dụng phải được đăng ký với một Microsoft Entra tenant. Khi đăng ký ứng dụng trong Azure portal, bạn chọn:

  • 🏠 Single tenant — chỉ có thể truy cập trong tenant của bạn
  • 🌍 Multi-tenant — có thể truy cập trong các tenant khác
📌 Lưu ý quan trọng

Khi đăng ký ứng dụng trong portal, một application object (instance duy nhất toàn cầu) và một service principal object sẽ được tự động tạo trong home tenant của bạn, cùng với một ID duy nhất toàn cầu (app ID / client ID).

Application Object

Một ứng dụng Microsoft Entra được giới hạn trong một application object duy nhất. Application object nằm trong Microsoft Entra tenant nơi ứng dụng được đăng ký (gọi là "home" tenant). Application object được dùng như template hoặc blueprint để tạo một hoặc nhiều service principal objects.

Application object mô tả 3 khía cạnh của ứng dụng:

  • Cách dịch vụ có thể phát hành token để truy cập ứng dụng
  • Tài nguyên mà ứng dụng có thể cần truy cập
  • Các hành động mà ứng dụng có thể thực hiện

Service Principal Object

Để truy cập tài nguyên được bảo mật bởi Microsoft Entra tenant, entity yêu cầu quyền truy cập phải được đại diện bởi security principal. Có 3 loại service principal:

LoạiMô tảĐặc điểm
Application Đại diện cục bộ của global application object trong một tenant/directory Được tạo trong mỗi tenant nơi ứng dụng được sử dụng; định nghĩa app có thể làm gì trong tenant cụ thể đó
Managed Identity Đại diện cho managed identity Cung cấp danh tính cho ứng dụng kết nối tới các tài nguyên hỗ trợ xác thực Microsoft Entra; có thể được cấp quyền truy cập nhưng không thể cập nhật trực tiếp
Legacy Đại diện cho ứng dụng cũ (tạo trước khi có app registrations) Có thể có credentials, SPNs, reply URLs; không có app registration liên kết

Mối quan hệ giữa Application Object và Service Principal:

  • Application object = đại diện toàn cầu (global) dùng cho tất cả tenants
  • Service principal = đại diện cục bộ (local) dùng cho tenant cụ thể
  • Quan hệ: Application object → 1:1 với ứng dụng phần mềm, 1:nhiều với service principal objects
  • Single-tenant app: chỉ có 1 service principal (trong home tenant)
  • Multi-tenant app: có service principal trong mỗi tenant mà user đã đồng ý

Module 3: Permissions & Consent

Quyền truy cập và quy trình đồng ý người dùng

Mô hình phân quyền OAuth 2.0

Microsoft Identity Platform triển khai giao thức ủy quyền OAuth 2.0. Các tài nguyên web-hosted tích hợp với nền tảng có resource identifier (application ID URI). Ví dụ:

  • Microsoft Graph: https://graph.microsoft.com
  • Microsoft 365 Mail API: https://outlook.office.com
  • Azure Key Vault: https://vault.azure.net

Trong OAuth 2.0, các tập hợp quyền này được gọi là scopes (hay permissions). Một permission được biểu diễn dưới dạng chuỗi. Ví dụ: https://graph.microsoft.com/Calendars.Read để đọc lịch người dùng.

Loại Permission

LoạiMô tảYêu cầu đồng ý
Delegated access Dùng bởi apps có user đang đăng nhập. App được ủy quyền để thực hiện thay mặt user đã đăng nhập User hoặc Administrator
App-only access Dùng bởi apps chạy không có user đăng nhập (background services, daemons) Chỉ Administrator

Các loại Consent

1. Static User Consent

Bạn phải chỉ định tất cả permissions cần thiết trong cấu hình app trong Azure portal. Nếu user (hoặc admin) chưa đồng ý, Microsoft Identity Platform sẽ nhắc user đồng ý.

⚠️ Hạn chế của Static Consent
  • App cần yêu cầu TẤT CẢ permissions ngay lần đăng nhập đầu tiên → danh sách dài làm nản lòng người dùng
  • App cần biết trước TẤT CẢ tài nguyên sẽ truy cập → khó tạo apps linh hoạt

2. Incremental and Dynamic User Consent

Với Microsoft Identity Platform endpoint, bạn có thể bỏ qua các static permissions đã định nghĩa trong Azure portal và yêu cầu permissions dần dần. Chỉ định scopes cần thiết bất kỳ lúc nào bằng cách thêm vào tham số scope khi yêu cầu access token.

⚠️ Quan trọng

Dynamic consent chỉ áp dụng cho delegated permissions, KHÔNG áp dụng cho app-only access permissions. Nếu cần quyền admin hoặc dùng dynamic consent, vẫn phải đăng ký TẤT CẢ permissions trong Azure portal.

3. Admin Consent

Bắt buộc khi app cần truy cập các high-privilege permissions. Admin consent đảm bảo administrators có quyền kiểm soát trước khi ủy quyền cho apps hoặc users truy cập dữ liệu nhạy cảm.

Ví dụ yêu cầu user consent:

HTTP
GET https://login.microsoftonline.com/common/oauth2/v2.0/authorize?
client_id=00001111-aaaa-2222-bbbb-3333cccc4444
&response_type=code
&redirect_uri=http%3A%2F%2Flocalhost%2Fmyapp%2F
&response_mode=query
&scope=
https%3A%2F%2Fgraph.microsoft.com%2Fcalendars.read%20
https%3A%2F%2Fgraph.microsoft.com%2Fmail.send
&state=12345

Module 4: Conditional Access

Kiểm soát truy cập theo điều kiện và chính sách

Conditional Access là gì?

Tính năng Conditional Access trong Microsoft Entra ID là một trong nhiều cách bảo mật app và dịch vụ. Conditional Access cho phép nhà phát triển và khách hàng doanh nghiệp bảo vệ dịch vụ theo nhiều cách:

  • 🔑 Xác thực đa yếu tố (Multifactor authentication - MFA)
  • 📱 Chỉ cho phép thiết bị đăng ký Intune truy cập dịch vụ cụ thể
  • 🌍 Hạn chế vị trí địa lý và dải IP của người dùng

Khi nào Conditional Access ảnh hưởng đến app?

Trong hầu hết các trường hợp thông thường, Conditional Access không thay đổi hành vi của app. Chỉ trong một số trường hợp khi app yêu cầu token gián tiếp hoặc âm thầm cho một dịch vụ, app mới cần thay đổi code để xử lý Conditional Access challenges.

Các scenarios yêu cầu xử lý Conditional Access:

  • Apps thực hiện on-behalf-of flow
  • Apps truy cập nhiều services/resources
  • Single-page apps dùng MSAL.js
  • Web apps gọi một resource

Ví dụ thực tế:

  • Scenario 1: App iOS single-tenant với Conditional Access policy → khi user đăng nhập, policy tự động được kích hoạt và user cần thực hiện MFA
  • Scenario 2: App dùng middle-tier service truy cập downstream API có policy → middle tier nhận "claims challenge" và phải chuyển lại cho client app để xử lý

Module 5: Microsoft Authentication Library (MSAL)

Thư viện xác thực cho ứng dụng Microsoft

MSAL là gì?

Microsoft Authentication Library (MSAL) cho phép nhà phát triển lấy security tokens từ Microsoft Identity Platform để xác thực người dùng và truy cập các web API bảo mật. MSAL hỗ trợ nhiều kiến trúc ứng dụng và nền tảng khác nhau.

Lợi ích của MSAL:

  • Không cần trực tiếp dùng OAuth libraries hoặc code theo protocol
  • Có thể lấy token thay mặt user hoặc application
  • Duy trì token cache và xử lý token refresh khi sắp hết hạn
  • Hỗ trợ chỉ định audience (personal Microsoft accounts, social identities, work/school accounts, sovereign clouds)
  • Cấu hình app từ configuration files
  • Hiển thị exceptions có thể hành động, logging và telemetry

Module 6: Shared Access Signatures (SAS) - Tổng quan

Ủy quyền truy cập giới hạn vào Azure Storage

SAS là gì?

Một shared access signature (SAS) là một URI đã ký chứa một token với tập hợp query parameters đặc biệt, cho biết cách client có thể truy cập tài nguyên. Chữ ký được xây dựng từ các tham số SAS và ký bằng khóa tạo ra SAS, được Azure Storage dùng để ủy quyền truy cập.

📌 Cấu trúc SAS URI

Ví dụ URI đầy đủ:

https://medicalrecords.blob.core.windows.net/patient-images/patient-116139-nq8z7f.jpg?sp=r&st=2020-01-20T11:42:32Z&se=2020-01-20T19:42:32Z&spr=https&sv=2019-02-02&sr=b&sig=SrW1HZ5Nb6MbRzTbXCaPm%2BJiSEn15tC91Y4umMPwVZs%3D

Cấu trúc SAS Token:

🔗 URI (Resource)
https://medicalrecords.blob.core.windows.net/patient-images/patient-116139-nq8z7f.jpg?
🔑 SAS Token — Query Parameters
sp=r — Quyền truy cập: r=read, w=write, d=delete, l=list, a=add, c=create
st=…Z — Thời gian bắt đầu (UTC)
se=…Z — Thời gian kết thúc (UTC)
spr=https — Giao thức: chỉ HTTPS
sv=2019-02-02 — Phiên bản Storage API
sr=b — Loại storage: b=blob, c=container, f=file, q=queue, t=table, s=share
sig=… — Chữ ký mật mã (HMAC-SHA256)

Phân tích chi tiết SAS Token components:

Tham sốMô tảGiá trị có thể
spQuyền truy cập (permissions)a=add, c=create, d=delete, l=list, r=read, w=write
stThời điểm bắt đầu (start time)Datetime UTC (ISO 8601)
seThời điểm hết hạn (expiry time)Datetime UTC (ISO 8601)
svPhiên bản API (service version)e.g., 2019-02-02
srLoại tài nguyên (resource type)b=blob, c=container, f=file, q=queue, s=share, t=table
sprGiao thức cho phéphttps, https,http
sigChữ ký mật mãBase64-encoded HMAC-SHA256

Ba loại SAS

⭐ USER DELEGATION SAS
Bảo mật: Microsoft Entra credentials — Chỉ áp dụng Blob Storage & Data Lake Storage. Khuyến nghị cao nhất — không cần lưu account key trong code.
SERVICE SAS
Bảo mật: Storage Account Key — Áp dụng Blob, Queue, Table, Azure Files. Phổ biến nhất.
ACCOUNT SAS
Bảo mật: Storage Account Key — Áp dụng tất cả storage services. Linh hoạt nhất nhưng quyền rộng nhất.
Loại SASBảo mật bằngPhạm vi áp dụngKhuyến nghị
User Delegation SASMicrosoft Entra credentials + permissions SASBlob Storage, Data Lake Storage✅ Khuyến nghị cao nhất

Best Practices khi dùng SAS

  • 🔐 Luôn dùng HTTPS để phân phối SAS, tránh tấn công man-in-the-middle
  • ⭐ SAS bảo mật nhất là User Delegation SAS — không cần lưu storage account key trong code
  • ⏰ Đặt thời gian hết hạn nhỏ nhất có thể — nếu SAS key bị lộ, thiệt hại chỉ trong thời gian ngắn
  • 📏 Áp dụng nguyên tắc ít quyền nhất (least privilege) — chỉ cấp quyền cần thiết
  • 🏢 Trong một số tình huống, tạo middle-tier service để quản lý user và quyền truy cập storage thay vì dùng SAS

Module 7: Khi nào dùng SAS

Lựa chọn đúng loại SAS cho từng tình huống

Kịch bản sử dụng SAS

Dùng SAS khi bạn muốn cung cấp quyền truy cập bảo mật vào tài nguyên trong storage account cho bất kỳ client nào không có quyền truy cập trực tiếp.

Kịch bản phổ biến: Dịch vụ nơi user đọc và ghi dữ liệu của họ vào storage account. Có 2 mẫu thiết kế điển hình:

Mẫu 1: Front-end proxy service

👤 Client
↕
🖥️ Front-end Proxy Service
(Authentication + Business Logic)
↕
☁️ Azure Storage
✅ Kiểm tra business rules ❌ Khó scale với large data / high-volume

Mẫu 2: Lightweight SAS provider service

1
Client → SAS ProviderYêu cầu SAS token (Lightweight Auth Service)
2
SAS Provider → ClientTrả về SAS token
3
Client → Azure Storage (trực tiếp)Truy cập thẳng Azure Storage bằng SAS — không qua proxy
✅ Không cần route ALL data qua proxy  ·  ✅ Dễ scale

Kịch bản yêu cầu SAS trong Copy Operations:

  • Copy blob sang blob ở storage account khác → cần SAS cho source blob (và tùy chọn destination)
  • Copy file sang file ở storage account khác → cần SAS cho source file
  • Copy blob sang file hoặc file sang blob → cần SAS cho source object (ngay cả khi cùng storage account)

Module 8: Stored Access Policies

Chính sách truy cập có thể tái sử dụng và thu hồi

Stored Access Policy là gì?

Stored access policy cung cấp một lớp kiểm soát bổ sung đối với service-level SAS ở phía server. Thiết lập stored access policy giúp nhóm các SAS và cung cấp thêm các ràng buộc cho signatures bị ràng buộc bởi policy.

Bạn có thể dùng stored access policy để thay đổi start time, expiry time, hoặc permissions của signature, hoặc thu hồi nó sau khi đã cấp.

📌 Tài nguyên hỗ trợ Stored Access Policies
  • Blob containers
  • File shares
  • Queues
  • Tables

Tạo Stored Access Policy

Access policy cho SAS gồm: start time, expiry time, và permissions. Bạn có thể chỉ định các tham số này trên signature URI, trong stored access policy, hoặc kết hợp cả hai. Tuy nhiên, không thể chỉ định một tham số ở cả hai nơi.

⚠️ Lưu ý thời gian kích hoạt

Khi thiết lập stored access policy trên container, table, queue hoặc share, có thể mất đến 30 giây để có hiệu lực. Trong thời gian này, requests đối với SAS liên kết với policy có thể thất bại với status code 403 (Forbidden).

Tạo Stored Access Policy bằng C# .NET:

JSON
BlobSignedIdentifier identifier = new BlobSignedIdentifier
{
    Id = "stored access policy identifier",
    AccessPolicy = new BlobAccessPolicy
    {
        ExpiresOn = DateTimeOffset.UtcNow.AddHours(1),
        Permissions = "rw"
    }
};

blobContainer.SetAccessPolicy(permissions: new BlobSignedIdentifier[] { identifier });

Tạo Stored Access Policy bằng Azure CLI:

CLI
az storage container policy create \
    --name <stored access policy identifier> \
    --container-name <container name> \
    --start <start time UTC datetime> \
    --expiry <expiry time UTC datetime> \
    --permissions <(a)dd, (c)reate, (d)elete, (l)ist, (r)ead, or (w)rite> \
    --account-key <storage account key> \
    --account-name <storage account name>

Sửa đổi hoặc Thu hồi Stored Access Policy

  • Sửa đổi: Gọi access control list operation để thay thế policy hiện có
  • Thu hồi bằng cách xóa: Gọi Set ACL với empty request body
  • Thu hồi bằng cách đổi tên: Thay đổi signed identifier → phá vỡ liên kết với SAS hiện có
  • Thu hồi bằng cách thay đổi expiry: Đặt expiry time về quá khứ → tất cả SAS liên kết hết hạn ngay lập tức

Xóa hoặc sửa đổi stored access policy sẽ ảnh hưởng ngay lập tức đến tất cả SAS liên kết với nó.

Module 10: Query Microsoft Graph qua REST

Gọi API Graph với HTTP và truy vấn OData

Cấu trúc REST Request

Microsoft Graph là một RESTful web API cho phép truy cập tài nguyên Microsoft Cloud. Để gọi một phương thức REST:

HTTP
{HTTP method} https://graph.microsoft.com/{version}/{resource}?{query-parameters}

Các thành phần của request:

Thành phầnMô tảVí dụ
{HTTP method}HTTP method của requestGET, POST, PATCH, PUT, DELETE
{version}Phiên bản APIv1.0 (production), beta (development)
{resource}Tài nguyên trong Microsoft Graphme, users, groups, drive, site
{query-parameters}OData query options tùy chọn$filter, $select, $orderby, $top

HTTP Methods:

MethodMục đíchRequest Body
GETĐọc dữ liệu từ resourceKhông cần
POSTTạo resource mới hoặc thực hiện actionJSON
PATCHCập nhật resource với giá trị mớiJSON
PUTThay thế resource bằng resource mớiJSON
DELETEXóa resourceKhông cần

Versions

VersionMục đíchKhuyến nghị
v1.0APIs thường được phát hành (Generally Available)✅ Dùng cho tất cả production apps
betaAPIs đang trong preview; có thể có breaking changes❌ KHÔNG dùng trong production apps

Query Parameters

Query parameters có thể là OData system query options hoặc các chuỗi khác để tùy chỉnh response.

Ví dụ filter:

HTTP
GET https://graph.microsoft.com/v1.0/me/messages?filter=emailAddress eq 'jon@contoso.com'

Công cụ hỗ trợ:

  • 🔧 Graph Explorer — https://developer.microsoft.com/graph/graph-explorer
  • 🔧 Postman — https://www.getpostman.com/
📌 Lưu ý về Resource Identifier

Trong requests tới Microsoft Identity Platform, nếu resource identifier bị bỏ qua trong scope parameter, resource sẽ được giả định là Microsoft Graph. Ví dụ: scope=User.Read tương đương với https://graph.microsoft.com/User.Read

Module 11: Microsoft Graph SDK

Tích hợp Graph nhanh hơn với .NET SDK

Giới thiệu Microsoft Graph SDK

Microsoft Graph SDKs được thiết kế để đơn giản hóa việc xây dựng ứng dụng chất lượng cao, hiệu quả và có khả năng phục hồi truy cập Microsoft Graph. SDK gồm hai thành phần:

  • 📦 Service library — Chứa models và request builders được tạo từ Microsoft Graph metadata, cung cấp trải nghiệm phong phú và có thể khám phá
  • ⚙️ Core library — Cung cấp tập hợp tính năng tăng cường làm việc với tất cả Microsoft Graph services: retry handling, secure redirects, transparent authentication, payload compression, pagination, batch requests

Tạo Microsoft Graph Client

// Multi-tenant apps can use "common",

Module 12: Best Practices Microsoft Graph

Tối ưu hiệu suất, batching và xử lý lỗi

Xác thực (Authentication)

Để truy cập dữ liệu trong Microsoft Graph, ứng dụng cần lấy OAuth 2.0 access token và trình bày cho Microsoft Graph theo một trong hai cách:

  • HTTP Authorization request header — dưới dạng Bearer token
  • Graph client constructor — khi sử dụng Microsoft Graph client library

Dùng MSAL API để lấy access token cho Microsoft Graph.

Consent và Authorization Best Practices

  • 🔐 Dùng ít quyền nhất — Chỉ yêu cầu permissions cần thiết và chỉ khi cần. Kiểm tra phần permissions trong method topics để chọn least privileged permissions.
  • 🎯 Dùng đúng loại permission theo scenario:
    • Interactive apps (có user đăng nhập) → dùng delegated permissions
    • Background services/daemons (không có user) → dùng application permissions
  • 👥 Cân nhắc trải nghiệm end user và admin — Ai đồng ý cho ứng dụng? Hiểu sự khác biệt giữa static, dynamic, và incremental consent.
  • 🌍 Cân nhắc multi-tenant applications — Khách hàng có thể có các chính sách kiểm soát ứng dụng và consent khác nhau. Xử lý 403 errors khi cần.
⚠️ Cảnh báo quan trọng

Dùng application permissions cho interactive scenarios có thể gây rủi ro compliance và security. Đảm bảo kiểm tra privileges của user để tránh họ truy cập thông tin không mong muốn.

Xử lý Responses hiệu quả

Pagination (Phân trang):

Khi query resource collections, Microsoft Graph có thể trả về kết quả theo nhiều trang do server-side page size limits. Ứng dụng phải luôn xử lý khả năng responses bị phân trang và dùng thuộc tính @odata.nextLink để lấy trang kết quả tiếp theo. Trang cuối không có thuộc tính @odata.nextLink.

Evolvable Enumerations:

Thêm members vào enumerations hiện có có thể làm hỏng ứng dụng đang dùng enums đó. Microsoft Graph API dùng evolvable enums để thêm members mới mà không gây breaking change. Mặc định, GET operation chỉ trả về known members. Có thể opt in nhận unknown members bằng HTTP Prefer request header.

Lưu trữ dữ liệu cục bộ

Ứng dụng nên thực hiện calls tới Microsoft Graph để lấy dữ liệu real-time khi cần. Chỉ cache hoặc lưu trữ dữ liệu cục bộ khi cần thiết cho scenario cụ thể, phải phù hợp với terms of use và privacy policy, và không vi phạm Microsoft APIs Terms of Use. Ứng dụng cũng phải triển khai proper retention và deletion policies.
