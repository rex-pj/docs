🔌 Phần 1: Khám Phá Dịch Vụ API Management

Phần 1: Khám Phá Dịch Vụ API Management

Azure API Management cung cấp chức năng cốt lõi để đảm bảo chương trình API thành công thông qua sự tham gia của nhà phát triển, thông tin kinh doanh, phân tích, bảo mật và bảo vệ. Mỗi API bao gồm một hoặc nhiều thao tác, và mỗi API có thể được thêm vào một hoặc nhiều sản phẩm. Để sử dụng API, các nhà phát triển đăng ký sản phẩm có chứa API đó, sau đó họ có thể gọi thao tác của API, tùy thuộc vào các chính sách sử dụng.

🌐 Client → API Gateway → Backend
Client (Người dùng) — gửi requests đến API Gateway endpoint
API Gateway (Cổng API) — xác thực, rate-limit, transform, cache, log
Backend APIs (Dịch vụ hậu) — nhận proxied requests, trả response
🔧 Management Plane
Cấu hình APIs, products, policies, subscriptions
Quản lý users, groups, roles, notifications
Tích hợp với Azure Portal, ARM, REST API
👩‍💻 Developer Portal (Cổng nhà phát triển)
Tự phục vụ — xem docs, test APIs, đăng ký subscriptions
Auto-generated từ API definitions (OpenAPI)
Có thể tùy chỉnh giao diện theo branding
📋 Các Tầng Dịch Vụ
Consumption — serverless, per-call billing
Developer — test/dev, không có SLA
Basic / Standard / Premium — production workloads

🏗️ Các Thành Phần API Management

Azure API Management bao gồm API gateway, management plane và developer portal. Các thành phần này được Azure lưu trữ và quản lý hoàn toàn theo mặc định.

1. API Gateway (Cổng API)

API Gateway là endpoint (điểm cuối) thực hiện các chức năng sau:

  • ✅ Chấp nhận lời gọi API và định tuyến đến backend phù hợp
  • ✅ Xác minh API key và các thông tin xác thực khác được gửi kèm yêu cầu
  • ✅ Thực thi hạn ngạch sử dụng và giới hạn tốc độ
  • ✅ Biến đổi yêu cầu và phản hồi theo các câu lệnh policy
  • ✅ Lưu cache phản hồi để cải thiện độ trễ và giảm tải cho backend
  • ✅ Phát ra logs, metrics và traces để giám sát, báo cáo và khắc phục sự cố

2. Management Plane (Mặt phẳng quản lý)

Đây là giao diện quản trị để thiết lập chương trình API. Sử dụng để:

  • 🔧 Cung cấp và cấu hình cài đặt dịch vụ API Management
  • 🔧 Định nghĩa hoặc nhập lược đồ API
  • 🔧 Đóng gói API vào các sản phẩm
  • 🔧 Thiết lập các policy như hạn ngạch hoặc biến đổi trên API
  • 🔧 Lấy thông tin từ phân tích
  • 🔧 Quản lý người dùng

3. Developer Portal (Cổng nhà phát triển)

Website được tạo tự động, có thể tùy chỉnh hoàn toàn với tài liệu API. Nhà phát triển có thể:

  • 📖 Đọc tài liệu API
  • 🧪 Gọi API qua console tương tác
  • 👤 Tạo tài khoản và đăng ký để lấy API key
  • 📊 Truy cập phân tích về mức sử dụng của mình
  • 📥 Tải xuống định nghĩa API
  • 🔑 Quản lý API key

📦 Sản Phẩm (Products)

Sản phẩm là cách các API được cung cấp cho nhà phát triển. Sản phẩm trong API Management có một hoặc nhiều API, và được cấu hình với tiêu đề, mô tả và điều khoản sử dụng.

Loại Sản PhẩmMô Tả
Open (Mở)Có thể sử dụng mà không cần đăng ký
Protected (Được bảo vệ)Phải đăng ký trước khi sử dụng. Phê duyệt có thể yêu cầu quản trị viên hoặc tự động phê duyệt

👥 Nhóm (Groups)

Nhóm được sử dụng để quản lý khả năng hiển thị của sản phẩm đối với nhà phát triển. API Management có các nhóm hệ thống bất biến sau:

NhómMô Tả
Administrators (Quản trị viên)Quản lý các instance dịch vụ API Management và tạo API, thao tác, sản phẩm. Quản trị viên đăng ký Azure là thành viên của nhóm này.
Developers (Nhà phát triển)Người dùng developer portal đã xác thực, xây dựng ứng dụng sử dụng API.
Guests (Khách)Người dùng developer portal chưa xác thực. Có thể được cấp quyền truy cập chỉ đọc nhất định.
💡 Nhóm tùy chỉnh

Ngoài các nhóm hệ thống, quản trị viên có thể tạo nhóm tùy chỉnh hoặc sử dụng nhóm bên ngoài trong Microsoft Entra tenant liên kết.

📋 Chính Sách (Policies)

Chính sách là tập hợp các câu lệnh được thực thi tuần tự trên yêu cầu hoặc phản hồi của API. Các câu lệnh phổ biến bao gồm chuyển đổi định dạng từ XML sang JSON và giới hạn tốc độ gọi.

Chính sách có thể được áp dụng ở các phạm vi khác nhau:

  • Global (Toàn cục): Áp dụng cho tất cả API
  • Product (Sản phẩm): Áp dụng cho một sản phẩm
  • API: Áp dụng cho một API cụ thể
  • Operation (Thao tác): Áp dụng cho một thao tác API

⚙️ Các Tầng Dịch Vụ API Management

Nhóm tầngMô tả
ClassicDeveloper, Basic, Standard, Premium tiers. Premium cho doanh nghiệp yêu cầu backend riêng tư, bảo mật nâng cao, triển khai đa vùng.
V2Bộ tầng mới với provisioning và scaling nhanh: Basic v2, Standard v2, Premium v2.
ConsumptionServerless gateway, scale theo nhu cầu, tính phí theo mỗi lần thực thi.

🚪 Phần 2: Khám Phá API Gateways

Phần 2: Khám Phá API Gateways

Giải pháp của bạn có thể chứa nhiều dịch vụ front-end và back-end. Trong tình huống này, client cần biết endpoint nào để gọi, và cần xử lý các mối lo ngại như SSL termination, xác thực, v.v.

API Management gateway (còn gọi là data plane hoặc runtime) là thành phần dịch vụ chịu trách nhiệm proxy API requests, áp dụng policies và thu thập telemetry.

❌ Không có API Gateway
1
Client gọi trực tiếp từng service
2
Service A — Auth + SSL + RateLimit
3
Service B — Auth + SSL + RateLimit
4
Service C — Auth + SSL + RateLimit
❌ Client code phức tạp — nhiều endpoints
❌ Coupling giữa client và backend services
❌ Mỗi service duplicate logic auth, SSL
✅ Với API Gateway
1
Client gọi 1 endpoint duy nhất
2
API Gateway — Auth, SSL, Rate Limit, Cache
3
Gateway proxy đến Service A / B / C
4
Services nhận requests đã được xử lý sẵn
✅ Client code đơn giản — 1 endpoint
✅ Tách biệt client và backend
✅ Cross-cutting concerns tập trung tại gateway

❓ Tại Sao Cần API Gateway?

Các vấn đề khi expose dịch vụ trực tiếp cho client:

  • ❌ Code client phức tạp: Client phải theo dõi nhiều endpoint và xử lý lỗi một cách bền vững
  • ❌ Tạo sự coupling: Client cần biết cách các dịch vụ được phân tách, khó bảo trì và refactor
  • ❌ Một thao tác cần nhiều service: Một yêu cầu có thể cần gọi nhiều dịch vụ
  • ❌ Mỗi service phải xử lý cross-cutting concerns: Auth, SSL, rate limiting, v.v.
  • ❌ Giới hạn giao thức: Service phải expose giao thức thân thiện với client như HTTP/WebSocket
  • ❌ Bề mặt tấn công: Service có endpoint công khai là mục tiêu tấn công tiềm năng

🏠 Gateway Managed và Self-hosted

API Management cung cấp cả gateway managed và self-hosted:

Loại GatewayMô TảKhi Nào Dùng
Managed (Được quản lý) Gateway mặc định được triển khai trong Azure cho mọi instance API Management trong mọi tầng dịch vụ. Tất cả lưu lượng API chạy qua Azure bất kể backend được lưu trữ ở đâu. Hầu hết các trường hợp thông thường khi backend trên Azure
Self-hosted (Tự lưu trữ) Phiên bản containerized tùy chọn của gateway managed mặc định. Được đóng gói dưới dạng Docker container dựa trên Linux, thường được triển khai lên Kubernetes. Hybrid và multicloud; backend on-premises; tuân thủ quy định địa phương
💡 Self-hosted Gateway

Self-hosted gateway cho phép khách hàng với cơ sở hạ tầng IT hybrid quản lý API được lưu trữ on-premises và trên nhiều đám mây từ một dịch vụ API Management duy nhất trong Azure. Nó thường được triển khai lên Azure Kubernetes Service và Azure Arc-enabled Kubernetes.

📜 Phần 3: Khám Phá Chính Sách API Management

Phần 3: Khám Phá Chính Sách API Management

Trong Azure API Management, các chính sách cho phép nhà xuất bản thay đổi hành vi của API thông qua cấu hình. Chính sách là tập hợp các câu lệnh được thực thi tuần tự trên yêu cầu hoặc phản hồi của API.

1
📥 INBOUND — Xử lý Request đến Xác thực subscription key · Kiểm tra rate limit · Thêm/sửa headers · Chuyển đổi request format
2
🔀 BACKEND — Chuyển tiếp đến Service Forward request tới backend URL · Thiết lập timeout · Load balancing · Circuit breaker
3
📤 OUTBOUND — Xử lý Response trả về Biến đổi response body · Lọc sensitive data · Thêm CORS headers · Cache response
4
⚠️ ON-ERROR — Xử lý Lỗi Bắt exception từ bất kỳ giai đoạn nào · Tùy chỉnh thông báo lỗi · Logging và alerting

📝 Hiểu Cấu Hình Chính Sách

Định nghĩa chính sách là tài liệu XML đơn giản mô tả chuỗi các câu lệnh inbound và outbound. Cấu hình được chia thành các phần inbound, backend, outbound và on-error.

XML
<policies>
  <inbound>
    <!-- statements to be applied to the request go here -->
  </inbound>
  <backend>
    <!-- statements to be applied before the request is forwarded to
         the backend service go here -->
  </backend>
  <outbound>
    <!-- statements to be applied to the response go here -->
  </outbound>
  <on-error>
    <!-- statements to be applied if there is an error condition go here -->
  </on-error>
</policies>
⚠️ Xử Lý Lỗi

Nếu xảy ra lỗi trong quá trình xử lý yêu cầu, các bước còn lại trong phần inbound, backend hoặc outbound sẽ bị bỏ qua và thực thi nhảy đến các câu lệnh trong phần on-error. Bằng cách đặt câu lệnh chính sách trong phần on-error, bạn có thể xem xét lỗi bằng thuộc tính context.LastError.

💻 Biểu Thức Chính Sách (Policy Expressions)

Biểu thức chính sách cung cấp phương tiện tinh vi để kiểm soát lưu lượng và sửa đổi hành vi API mà không cần viết code chuyên biệt. Biểu thức chính sách là:

  • Câu lệnh C# đơn: được đặt trong @(expression)
  • Khối code C# nhiều câu lệnh: được đặt trong @{expression} và phải trả về giá trị

Ví dụ sử dụng biểu thức chính sách và policy set-header để thêm dữ liệu người dùng vào request đến:

XML
<policies>
    <inbound>
        <base />
        <set-header name="x-request-context-data" exists-action="override">
            <value>@(context.User.Id)</value>
            <value>@(context.Deployment.Region)</value>
        </set-header>
    </inbound>
</policies>

📐 Áp Dụng Chính Sách Ở Các Phạm Vi Khác Nhau

Nếu bạn có chính sách ở cấp global và chính sách được cấu hình cho một API, thì mỗi khi API đó được sử dụng, cả hai chính sách đều được áp dụng. API Management cho phép sắp xếp xác định của các câu lệnh chính sách kết hợp thông qua phần tử base.

XML
<policies>
    <inbound>
        <cross-domain />
        <base />
        <find-and-replace from="xyz" to="abc" />
    </inbound>
</policies>

Trong ví dụ trên:

  1. Câu lệnh cross-domain sẽ thực thi trước
  2. Câu lệnh find-and-replace sẽ thực thi sau bất kỳ chính sách nào ở phạm vi rộng hơn

🔍 Lọc Nội Dung Phản Hồi (Filter Response Content)

Ví dụ sau minh họa cách lọc các phần tử dữ liệu từ payload phản hồi dựa trên sản phẩm liên kết với yêu cầu. Đoạn code giả định nội dung phản hồi định dạng JSON chứa các thuộc tính minutely, hourly, daily và flags:

XML
<policies>
  <inbound>
    <base />
  </inbound>
  <backend>
    <base />
  </backend>
  <outbound>
    <base />
    <choose>
      <when condition="@(context.Response.StatusCode == 200 && context.Product.Name.Equals("Starter"))">
        <set-body>
          @{
            var response = context.Response.Body.As<JObject>();
            foreach (var key in new [] {"minutely", "hourly", "daily", "flags"}) {
              response.Property(key).Remove();
            }
            return response.ToString();
          }
        </set-body>
      </when>
    </choose>
  </outbound>
  <on-error>
    <base />
  </on-error>
</policies>

⚡ Phần 4: Tạo Chính Sách Nâng Cao

Phần 4: Tạo Chính Sách Nâng Cao

Phần này cung cấp tài liệu tham khảo cho các chính sách API Management nâng cao sau:

  • Control flow: Áp dụng câu lệnh chính sách có điều kiện dựa trên kết quả đánh giá biểu thức Boolean
  • Forward request: Chuyển tiếp yêu cầu đến dịch vụ backend
  • Limit concurrency: Ngăn các chính sách được thực thi bởi nhiều hơn số yêu cầu được chỉ định cùng một lúc
  • Log to Event Hubs: Gửi tin nhắn theo định dạng được chỉ định đến event hub
  • Mock response: Hủy bỏ thực thi pipeline và trả về phản hồi giả trực tiếp cho caller
  • Retry: Thực thi lại các câu lệnh chính sách con cho đến khi điều kiện được đáp ứng
  • Return response: Hủy bỏ thực thi pipeline và trả về phản hồi tùy chỉnh

🔀 Control Flow (Luồng điều khiển)

Chính sách choose áp dụng các câu lệnh chính sách được đặt trong nó dựa trên kết quả đánh giá biểu thức boolean, tương tự như cấu trúc if-then-else hoặc switch trong ngôn ngữ lập trình.

XML
<choose>
    <when condition="Boolean expression | Boolean constant">
        <!-- một hoặc nhiều câu lệnh chính sách áp dụng khi điều kiện trên là true -->
    </when>
    <when condition="Boolean expression | Boolean constant">
        <!-- một hoặc nhiều câu lệnh chính sách áp dụng khi điều kiện trên là true -->
    </when>
    <otherwise>
        <!-- một hoặc nhiều câu lệnh chính sách áp dụng nếu không có điều kiện nào ở trên là true -->
    </otherwise>
</choose>
💡 Lưu ý

Chính sách control flow phải chứa ít nhất một phần tử <when/>. Phần tử <otherwise/> là tùy chọn. Điều kiện trong các phần tử <when/> được đánh giá theo thứ tự xuất hiện.

➡️ Forward Request (Chuyển tiếp yêu cầu)

Chính sách forward-request chuyển tiếp yêu cầu đến dịch vụ backend được chỉ định trong request context. URL dịch vụ backend được chỉ định trong cài đặt API.

XML
<forward-request timeout="time in seconds" follow-redirects="true | false"/>
⚠️ Quan trọng

Việc xóa policy này dẫn đến yêu cầu không được chuyển tiếp đến dịch vụ backend. Các chính sách trong phần outbound được đánh giá ngay sau khi hoàn thành thành công các chính sách trong phần inbound.

🚦 Limit Concurrency (Giới hạn đồng thời)

Chính sách limit-concurrency ngăn các chính sách được đặt trong nó được thực thi bởi nhiều hơn số yêu cầu được chỉ định tại bất kỳ thời điểm nào. Khi vượt quá số đó, các yêu cầu mới thất bại ngay lập tức với mã trạng thái 429 Too Many Requests.

XML
<limit-concurrency key="expression" max-count="number">
    <!-- nested policy statements -->
</limit-concurrency>

📡 Log to Event Hubs (Ghi log đến Event Hubs)

Chính sách log-to-eventhub gửi tin nhắn theo định dạng được chỉ định đến event hub được xác định bởi entity Logger. Policy được sử dụng để lưu thông tin context yêu cầu/phản hồi được chọn để phân tích trực tuyến hoặc ngoại tuyến.

XML
<log-to-eventhub logger-id="id of the logger entity"
    partition-id="index of the partition where messages are sent"
    partition-key="value used for partition assignment">
  Expression returning a string to be logged
</log-to-eventhub>

🎭 Mock Response (Phản hồi giả)

Chính sách mock-response được sử dụng để mock API và thao tác. Nó hủy bỏ thực thi pipeline thông thường và trả về phản hồi giả cho caller. Policy luôn cố gắng trả về phản hồi có độ trung thực cao nhất:

  • Ưu tiên các ví dụ nội dung phản hồi khi có
  • Tạo phản hồi mẫu từ schema khi được cung cấp
  • Trả về phản hồi không có nội dung nếu không tìm thấy ví dụ hoặc schema
XML
<mock-response status-code="code" content-type="media type"/>

🔄 Retry (Thử lại)

Chính sách retry thực thi các chính sách con của nó một lần và sau đó thử lại cho đến khi điều kiện condition của retry trở thành false hoặc count retry cạn kiệt.

XML
<retry
    condition="boolean expression or literal"
    count="number of retry attempts"
    interval="retry interval in seconds"
    max-interval="maximum retry interval in seconds"
    delta="retry interval delta in seconds"
    first-fast-retry="boolean expression or literal">
    <!-- One or more child policies. No restrictions -->
</retry>

↩️ Return Response (Trả về phản hồi)

Chính sách return-response hủy bỏ thực thi pipeline và trả về phản hồi mặc định hoặc tùy chỉnh cho caller. Phản hồi mặc định là 200 OK không có body.

XML
<return-response response-variable-name="existing context variable">
  <set-header/>
  <set-body/>
  <set-status/>
</return-response>

Phần 5: Bảo Mật API với Azure API Management

🔑 Subscriptions (Đăng ký)

Trong Azure API Management, subscriptions là cách phổ biến nhất để người tiêu thụ API truy cập các API được xuất bản. Bằng cách xuất bản API qua API Management, bạn có thể dễ dàng bảo mật quyền truy cập API bằng subscription key.

Nhà phát triển cần đưa vào subscription key hợp lệ trong các HTTP request khi gọi API. Nếu không có key hợp lệ, các lệnh gọi sẽ:

  • ❌ Bị từ chối ngay lập tức bởi API Management gateway
  • ❌ Không được chuyển tiếp đến dịch vụ backend

Cách sử dụng Subscription Key

Người đăng ký có thể sử dụng subscription key theo hai cách:

  • HTTP Header: Thêm header Ocp-Apim-Subscription-Key vào request
  • Query Parameter: Thêm parameter subscription-key vào URL

Phạm vi Subscriptions

Phạm viMô tả
Product scopeLiên kết với một sản phẩm cụ thể
All APIs scopeCấp quyền truy cập tất cả API trong instance
Single API scopeCấp quyền truy cập một API cụ thể
All-access subscriptionSubscription tích hợp cấp quyền truy cập tất cả API (chỉ dùng cho test/debug)
⚠️ Cảnh Báo Bảo Mật

All-access subscription cho phép truy cập mọi API trong instance. Không bao giờ nhúng all-access subscription key vào client app hoặc sử dụng cho quyền truy cập API thường xuyên.

Chính Sách Bảo Mật Mẫu - Validate JWT

XML
<policies>
    <inbound>
        <validate-jwt header-name="Authorization"
                      failed-validation-httpcode="401"
                      failed-validation-error-message="Unauthorized">
            <openid-config url="https://login.microsoftonline.com/{tenantId}/.well-known/openid-configuration" />
            <required-claims>
                <claim name="aud">
                    <value>{app-id}</value>
                </claim>
            </required-claims>
        </validate-jwt>
        <base />
    </inbound>
</policies>

Chính Sách Kiểm Tra Client Certificate

XML
<policies>
    <inbound>
        <choose>
            <when condition="@(context.Request.Certificate == null ||
                              !context.Request.Certificate.Verify() ||
                              context.Request.Certificate.Thumbprint != 'expected-thumbprint')">
                <return-response>
                    <set-status code="403" reason="Invalid client certificate" />
                </return-response>
            </when>
        </choose>
        <base />
    </inbound>
</policies>

Phần 6: Biểu Thức Chính Sách API Management

Biểu thức chính sách sử dụng cú pháp C# 7. Mỗi biểu thức có quyền truy cập vào biến context được cung cấp ngầm định và một tập con được phép của các kiểu .NET Framework.

📌 Cú Pháp

  • Biểu thức đơn câu lệnh: Đặt trong @(expression), là câu lệnh C# hợp lệ
  • Biểu thức nhiều câu lệnh: Đặt trong @{expression}. Tất cả các đường code phải kết thúc bằng câu lệnh return

Ví dụ biểu thức:

XML
@(true)

@((1+1).ToString())

@("Hi There".Length)

@(Regex.Match(context.Response.Headers.GetValueOrDefault("Cache-Control",""),
  @"max-age=(?<maxAge>\d+)").Groups["maxAge"]?.Value)

@(context.Variables.ContainsKey("maxAge") ? int.Parse((string)context.Variables["maxAge"]) : 3600)

@{
  string[] value;
  if (context.Request.Headers.TryGetValue("Authorization", out value))
  {
      if(value != null && value.Length > 0)
      {
          return Encoding.UTF8.GetString(Convert.FromBase64String(value[0]));
      }
  }
  return null;
}

🔐 Các Phương Thức Đặc Biệt

Xử lý JWT:

XML
// Parse JWT token
Jwt AsJwt(input: this string)
bool TryParseJwt(input: this string, result: out Jwt)

// Jwt properties:
// - Algorithm, Audiences, Claims, ExpirationTime, Id, Issuer, Subject, Type

// Example: Kiểm tra audience trong JWT
@(context.Request.Headers.GetValueOrDefault("Authorization","")
  .AsJwt()?.Claims["aud"].FirstOrDefault() == "myapi")

Xử lý Basic Auth:

XML
// Parse Basic Auth credentials
BasicAuthCredentials AsBasic(input: this string)

// Example: Lấy thông tin đăng nhập Basic Auth
@{
    var basic = context.Request.Headers.GetValueOrDefault("Authorization","")
                .AsBasic();
    return basic?.UserId;
}

Mã hóa/Giải mã:

XML
// Encrypt
byte[] Encrypt(input: byte[], alg: string, key: byte[], iv: byte[])

// Decrypt
byte[] Decrypt(input: byte[], alg: string, key: byte[], iv: byte[])

// Verify certificate (không kiểm tra revocation)
bool VerifyNoRevocation(input: X509Certificate2)

📝 Ví Dụ Thực Tế

Ví dụ 1: Rate limiting theo subscription

XML
<rate-limit-by-key calls="10" renewal-period="60"
    counter-key="@(context.Subscription.Id)" />

Ví dụ 2: Thêm correlation ID

XML
<set-header name="x-correlation-id" exists-action="skip">
    <value>@(context.RequestId.ToString())</value>
</set-header>

Ví dụ 3: Kiểm tra role từ JWT claims

XML
<choose>
    <when condition="@{
        var jwt = context.Request.Headers.GetValueOrDefault("Authorization","")
                         .AsJwt();
        return jwt != null && jwt.Claims["roles"]
                .Any(r => r == "admin");
    }">
        <!-- Allow admin access -->
    </when>
    <otherwise>
        <return-response>
            <set-status code="403" reason="Forbidden" />
        </return-response>
    </otherwise>
</choose>

📋 Tóm Tắt Quan Trọng Cho Kỳ Thi

✅ Các điểm cần nhớ
  • API Management có 3 thành phần chính: Gateway, Management Plane, Developer Portal
  • Policy được cấu trúc trong 4 section: inbound, backend, outbound, on-error
  • Biểu thức đơn: @(expression); Nhiều câu lệnh: @{...return value;}
  • Biến context có sẵn trong mọi biểu thức chính sách
  • Subscription key được gửi qua header Ocp-Apim-Subscription-Key hoặc query param subscription-key
  • Gateway có hai loại: Managed (cloud) và Self-hosted (containerized, Docker/Kubernetes)
  • Policy được áp dụng ở 4 phạm vi: global, product, API, operation
  • Phần tử <base /> xác định vị trí thực thi policy từ phạm vi cha
